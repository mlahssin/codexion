Course 8 of the Codexion series

Timing

Codexion is a real-time problem wearing a concurrency costume. Every requirement — burn out at exactly this deadline, log it within 10ms, hold the dongles for precisely this long, don't grant one until its cooldown elapses — is a claim about time. This course covers where timestamps actually come from, why one clock can run backwards and the other can't, why a sleep is a floor and never a ceiling, and why the gap between your thread becoming runnable and actually running is the real limit on precision. It ends where the subject's hardest requirement lives: detecting burnout within 10ms.

Chapter 01

Clocks: gettimeofday() & clock_gettime()

1.1  Where "what time is it" actually comes from

Why this exists Your CPU has no idea what time it is. What it has is a counter — most commonly the x86 TSC (Time Stamp Counter), a register that increments at a known rate. The kernel calibrates that counter against a reference, tracks an offset, and turns "counter value" into "seconds since the epoch." Every timestamp in your Codexion log traces back through that translation.

The kernel exposes several clock sources (TSC, HPET, ACPI PM timer) and picks the best available — you can see the choice on your machine:

cat /sys/devices/system/clocksource/clocksource0/current_clocksource

1.2  The two APIs

// gettimeofday — the older POSIX interface, microsecond resolution
struct timeval tv;
gettimeofday(&tv, NULL);
long ms = tv.tv_sec * 1000L + tv.tv_usec / 1000;

// clock_gettime — the modern one, nanosecond resolution, selectable clock
struct timespec ts;
clock_gettime(CLOCK_MONOTONIC, &ts);
long ms = ts.tv_sec * 1000L + ts.tv_nsec / 1000000;

Resolution is not accuracy clock_gettime reports nanoseconds. That does not mean it's accurate to a nanosecond — the underlying counter has its own granularity, and reading the clock itself costs time. Reporting a value with nine decimal places says nothing about how many of them are meaningful. For Codexion's millisecond-scale timings, both APIs are far more precise than you need; the real limits come from scheduling (Chapter 5), not the clock.

1.3  The vDSO: why reading the clock is nearly free

Asking the kernel for the time sounds like it should be a syscall — a mode switch, hundreds of nanoseconds. If your monitor thread polls burnout deadlines frequently, that cost would matter. So Linux cheats, legitimately: the vDSO (virtual dynamic shared object) is a small page the kernel maps into every process, containing both the code and the current timekeeping data needed to compute the time in userspace.

So clock_gettime(CLOCK_MONOTONIC, …) typically executes entirely in your process — read the TSC, apply the kernel-maintained scaling factors from the shared page, return. No syscall, no mode switch, tens of nanoseconds.

Connect to Codexion This is why "my monitor thread checks the time constantly, isn't that expensive?" has a reassuring answer: reading the clock is one of the cheapest things you can do. Your monitor's cost is dominated by how often it sleeps and wakes (Chapter 4), not by asking the time.

Chapter 02

Monotonic vs. realtime clocks

2.1  Wall-clock time can move backwards

Why this exists CLOCK_REALTIME answers "what time is it in the world?" — and the world's answer can change. NTP adjusts it. An administrator can set it. Daylight saving, leap seconds, a VM resuming from a snapshot. It can jump forward, and critically, it can jump backward. Any code that computes a duration by subtracting two realtime readings is one ntpd correction away from a negative elapsed time.
CLOCK_REALTIMECLOCK_MONOTONIC

2.2  The rule

The one-liner worth memorizing Use REALTIME for a point in the world. Use MONOTONIC for an interval. Every Codexion timing question — how long since this coder started compiling, how long until its burnout deadline, has the cooldown elapsed — is an interval question.

Consider what a backward jump does to Codexion. A coder started compiling at wall-clock t=1000. Mid-compile, NTP corrects the clock back 2 seconds. Now now() - last_compile_start is negative. Your burnout check compares against a deadline that's suddenly 2 seconds further away than it should be. Or worse, depending on your arithmetic and types, an unsigned subtraction underflows into an enormous positive number and every coder appears to have burned out simultaneously.

2.3  The awkward part: what the subject allows

Codexion's allowed external functions include gettimeofday — which is always wall-clock — and not clock_gettime. And the subject says outright: "real-time measurements using gettimeofday() are acceptable and recommended for simplicity."

Connect to Codexion — how to handle this honestly Use gettimeofday. The subject recommends it, it's in the allowed list, and over a simulation lasting seconds on a machine that isn't being reconfigured mid-run, the risk of an NTP step is negligible. But know that it's a wall-clock source and that this would be the wrong choice in production code measuring intervals. If an evaluator asks "why not a monotonic clock?", the strong answer is: "Because the subject specifies gettimeofday and it's sufficient for a short-lived simulation — but in production I'd use CLOCK_MONOTONIC, because gettimeofday can step backward under NTP and every timing question here is really an interval question." That answer shows you know the tradeoff rather than being unaware of it.

One related note: pthread_cond_timedwait defaults to CLOCK_REALTIME for its absolute deadline — consistent with gettimeofday, so at least your two time sources agree.

Chapter 03

Timer interrupts & hrtimers

3.1  How the kernel regains control

Why this exists A running thread has the CPU. It's executing your instructions, not the kernel's. So how does the kernel ever preempt it (Course 3)? It can't — not by software. Something external must interrupt the CPU. That something is a hardware timer, programmed to fire an interrupt at a chosen moment, forcibly transferring control to the kernel's handler regardless of what the running thread wanted.

Every preemption, every expiring usleep, every pthread_cond_timedwait timeout ultimately rides on a hardware timer interrupt. It is the only mechanism by which "later, do this" becomes real.

3.2  From periodic ticks to tickless

Older kernels programmed the timer to fire at a fixed rate — CONFIG_HZ, often 100, 250, or 1000 times per second — and did all timekeeping and scheduling work on those ticks. That has an obvious flaw: timer granularity is capped at one tick. With HZ=250, a 1ms sleep can't be honored more precisely than 4ms.

Modern Linux uses hrtimers (high-resolution timers) and a tickless design: instead of a fixed drumbeat, the kernel keeps pending timer expirations in a red-black tree ordered by deadline, and programs the hardware to fire exactly when the next one is due. Fewer interrupts when idle, far better granularity when needed.

The pattern, again A red-black tree ordered by expiration time, so "what's next?" is the leftmost node. This is the third time this shape has appeared: CFS's runqueue (Course 3), the hrtimer queue (here), and your Codexion request heap. Whenever a system must repeatedly answer "which pending thing is soonest?", it reaches for the same structure. That's not a coincidence you should let pass at defense.

3.3  What this bounds

Hrtimers give sub-millisecond granularity — comfortably better than Codexion's 10ms burnout requirement. So the timer subsystem is not your limiting factor. The limit is what happens after the interrupt fires: the kernel must mark your thread runnable, and then the scheduler must actually give it a core. That gap is scheduler latency (Chapter 5), and it's where the real uncertainty lives.

Chapter 04

usleep() and why sleep is never exact

4.1  What usleep actually promises

Read the contract carefully usleep(200000) does not promise "resume in exactly 200ms." It promises "sleep for at least 200ms." The distinction is the whole chapter. Sleep functions have a lower bound and no upper bound — a sleep may overshoot arbitrarily, and nothing in POSIX says otherwise.

Mechanically: usleep arms an hrtimer, marks your thread TASK_INTERRUPTIBLE, removes it from the runqueue, and calls schedule(). When the timer fires, the kernel marks you TASK_RUNNING and puts you back on a runqueue — runnable, not running. You now wait, again, for a core.

4.2  Where the overshoot comes from

Timer fires on time. Your thread runs later. The gap is where the imprecision lives.

Contributing factors, roughly in order of typical magnitude:

  • Scheduler latency — other runnable threads are ahead of you; you wait for a core (Chapter 5).
  • Timer granularity — sub-millisecond on modern hrtimers; usually the smallest term.
  • Wakeup + context switch cost — microseconds.
  • System load — under heavy contention, the wait for a core dominates everything else.

4.3  The pattern for accurate durations

Because sleeps overshoot, computing your next deadline by adding to now accumulates drift:

// DRIFTS — each overshoot compounds into the next cycle
while (running) {
    usleep(200000);
    do_thing();          // by cycle 50, you're tens of ms late
}

// CORRECT — deadlines computed from a fixed origin, drift doesn't accumulate
long next = start_ms + 200;
while (running) {
    long now = now_ms();
    if (now < next) usleep((next - now) * 1000);
    do_thing();
    next += 200;         // absolute schedule, immune to per-sleep overshoot
}
Connect to Codexion Your coder simulates compiling with usleep(time_to_compile * 1000). A few hundred microseconds of overshoot per phase is fine — the evaluation scale explicitly allows "a small timing tolerance" and asks evaluators to "discuss borderline cases honestly." But burnout deadlines must be computed from last_compile_start as an absolute point, exactly as the subject defines them (last_compile_start + time_to_burnout), never by accumulating sleep durations. The subject's own formula is telling you to use the absolute pattern.

Chapter 05

Scheduler latency

5.1  The gap between runnable and running

Why this exists Scheduler latency is the delay between a thread becoming runnable and it actually running on a core. It is not a bug or a defect — it is the unavoidable consequence of having more runnable threads than cores. It's also the single largest source of timing imprecision in Codexion, and the reason the subject writes "allow a minimal tolerance when testing."

On an idle machine, latency is microseconds. Under load — say, 200 coder threads all becoming runnable when a dongle frees — it can be milliseconds. That range spans Codexion's entire 10ms budget.

5.2  What makes it worse

FactorEffect
More runnable threads than coresDirectly increases queueing delay
Thundering herd (Course 6)N threads become runnable at once; N−1 waste their turn re-checking and sleeping
Other system loadYour threads compete with everything else on the machine
Running under Valgrind10–50× slowdown; timings become meaningless
Connect to Codexion — a real trap Running valgrind --tool=helgrind ./codexion 5 500 200 200 200 10 0 fifo will produce burnouts at wildly wrong times, or burnouts in configurations that should be feasible. That is expected and not a bug in your code — Valgrind serializes threads onto a single virtual core and slows everything by an order of magnitude. Use Helgrind to check for races and lock-order violations; use a native build to check timing. Never judge one with the other. Say this at defense if the evaluator runs Helgrind and points at the timestamps.

5.3  Designing around it

  • Don't fight it. You cannot make CFS give you a core faster. Accept a tolerance and design so a few hundred microseconds don't matter.
  • Reduce runnable threads. Blocked threads don't compete. This is another argument for condition variables over polling (Course 6) — a polling coder is runnable and in the way; a blocked one isn't.
  • Prefer signal over broadcast where correct, to avoid manufacturing your own thundering herd.
  • Never hold a mutex while sleeping unnecessarily — the deliberate dongle-possession case (Course 5) excepted.
What "real-time" would require Genuine bounded latency needs SCHED_FIFO/SCHED_DEADLINE, usually a PREEMPT_RT kernel, and often CPU isolation. None of that is available to you here — sched_setscheduler isn't in the allowed function list, and the subject doesn't ask for it. Codexion is a soft real-time exercise: it wants you to understand deadlines and build a deadline-aware scheduler, while acknowledging in its own text that the underlying OS gives no hard guarantee.

Chapter 06

Burnout detection

6.1  What the subject demands

Three requirements, from the subject and the evaluation scale:

  • A coder burns out if it hasn't started compiling within time_to_burnout ms of its last compile start (or of the simulation start).
  • "A separate monitor thread must detect burnout precisely and stop the simulation."
  • The burnout log must appear within 10ms of the actual burnout time — and per the scale, it must be the last line printed.

So each coder's deadline is last_compile_start + time_to_burnout — the same value edf uses as its priority key. One field, two purposes.

6.2  Three monitor designs

DesignPrecisionCostVerdict
Busy-spin — loop reading the clock, never sleepExcellent (µs)Burns an entire core continuouslyWasteful; steals a core from the coders and worsens their latency
Fixed poll — usleep(1000), check all coders, repeat~1ms + latency — well inside budget~1000 cheap wakeups/secSimple, robust, entirely sufficient
Deadline-driven — cond_timedwait until the earliest deadlineExcellentMinimal — wakes only when neededElegant; needs care when deadlines change under you
Why the middle option is usually right A 1ms poll gives you a 1ms detection granularity plus scheduler latency — comfortably inside 10ms even under load — and reading the clock is nearly free thanks to the vDSO (Chapter 1). The deadline-driven design is more sophisticated but introduces a real subtlety: a coder starting a compile changes its deadline, so your monitor's sleep target may become stale and must be re-signalled. Choose either, but choose it deliberately and be able to say why.

6.3  The shape of a correct monitor

void *monitor(void *arg) {
    sim_t *sim = arg;
    while (1) {
        pthread_mutex_lock(&sim->mtx);

        // 1. Did everyone finish? (checked under the same lock)
        if (sim->coders_done == sim->n) { sim->stop = 1; pthread_cond_broadcast(&sim->cv);
            pthread_mutex_unlock(&sim->mtx); break; }

        // 2. Has anyone burned out?
        long now = now_ms();
        for (int i = 0; i < sim->n; i++) {
            if (now >= sim->coders[i].last_compile_start + sim->time_to_burnout) {
                sim->stop = 1;                    // set BEFORE printing
                log_burnout(sim, i, now);         // last line, under the log mutex
                pthread_cond_broadcast(&sim->cv); // wake every sleeper so join() works
                pthread_mutex_unlock(&sim->mtx);
                return NULL;
            }
        }
        pthread_mutex_unlock(&sim->mtx);
        usleep(1000);                              // sleep OUTSIDE the lock
    }
    return NULL;
}

Four details carry the correctness:

  • Reads are under the mutex. last_compile_start is written by coder threads; reading it unlocked is a data race Helgrind will find (Course 9).
  • The sleep is outside the mutex. Sleeping while holding the lock would block every coder for 1ms out of every 1ms.
  • Stop is set before the burnout line is printed, and coders check stop before printing — that's what makes burnout the last line.
  • Broadcast, not signal. Every blocked coder must wake to observe stop, or pthread_join hangs (Course 6, Chapter 3).
Connect to Codexion — the ordering is the exam The infeasible benchmark (./codexion 5 500 200 200 200 10 0 fifo) exists to test exactly this. Burnout is guaranteed; what's being graded is whether your burnout line lands within 10ms and whether anything prints after it. Set the flag, print, broadcast, in that order, all under the lock — and have every coder check stop before every log line. Course 10 builds the full architecture.
