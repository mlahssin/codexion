Debugging threads

Concurrency breaks the assumption every debugging habit rests on: that the same input produces the same execution. A race can hide through ten thousand clean runs and surface during your defense. A printf added to investigate can make it disappear. This course covers the tools that reason about your code's structure rather than one lucky run's outcome — GDB for problems that happened, Helgrind and TSan for problems that haven't yet — and ends with the exact checklist an evaluator will run against you.

Chapter 01

Why threads break debugging

1.1  The reproducibility problem

Why this exists Single-threaded debugging rests on an assumption so basic you never notice it: same input, same execution. Set a breakpoint, run it again, you land in the same place with the same state. Concurrency destroys that. The interleaving of your five coder threads is chosen by the scheduler, influenced by system load, cache state, and luck. The same binary with the same arguments can take a different path every single run.

The consequences are specific and brutal:

  • A bug that appears once in 10,000 runs is not reproducible on demand — you can't step through it.
  • Attaching a debugger changes the timing, so the bug often stops appearing (a "heisenbug").
  • Adding a printf to investigate serializes threads slightly and can make the bug vanish.
  • Passing every test proves nothing. The race was there; the interleaving just never came up.

1.2  Why the evaluation scale doesn't trust you

Read what the scale actually instructs evaluators to do:

From the evaluation scale "Since this is a concurrency project, you must also check for data races and deadlocks, which are not always visible by simply reading the logs. Use the thread-specific tooling available on the computer… Do not simply trust the evaluated student's explanation that the code is thread-safe."

That's not skepticism about you personally. It's an acknowledgement that nobody can prove thread-safety by observation. Correct-looking output is compatible with a fatal race. The scale requires tools because tools reason about the structure of your synchronization, not the outcome of one lucky run.

1.3  The right mental model

The shift Stop asking "did it produce the right answer?" Start asking "could any interleaving produce a wrong answer?" The first question is answered by running the program. The second is answered by analysis — either your own reasoning about happens-before relationships (Course 4), or a tool that does that reasoning for you. Only the second question is the one that matters.
Connect to Codexion This reframing is the whole course. You will use GDB for the crashes and hangs — problems with a location. You will use Helgrind, DRD, and ThreadSanitizer for the races — problems with no location, that may not have happened yet, and that the scale will find whether or not you did.

Chapter 02

GDB with threads

2.1  The essential commands

GDB knows about threads. Compile with -g (keep -Wall -Wextra -Werror -pthread) and these become your core vocabulary:

CommandWhat it does
info threadsList every thread, its ID, and where it currently is
thread 3Switch context to thread 3
thread apply all btBacktrace every thread — the single most valuable command for a hang
btBacktrace of the current thread
frame 2 / info localsInspect a specific stack frame's variables
set scheduler-locking onOnly the current thread runs while stepping — useful, but changes timing drastically
break codexion.c:88 thread 3Breakpoint that fires only for thread 3

2.2  Diagnosing a hang

The technique that solves deadlocks Your program hangs. Don't restart it — attach to it and look at every thread at once. A deadlock is a cycle, and a cycle is visible the instant you can see all the backtraces side by side.
$ ./codexion 5 2000 200 200 200 10 0 fifo &
# it hangs...
$ gdb -p $(pgrep codexion)
(gdb) thread apply all bt
Thread 3 (coder 2):
#0  futex_wait (...) at ...
#1  __pthread_mutex_lock (mutex=0x5555555592a0) at ...
#2  take_dongle (d=0x5555555592a0) at codexion.c:88     ← waiting on dongle 3
#3  coder_routine (arg=0x555555559100) at codexion.c:141

Thread 4 (coder 3):
#0  futex_wait (...) at ...
#1  __pthread_mutex_lock (mutex=0x5555555592e0) at ...
#2  take_dongle (d=0x5555555592e0) at codexion.c:88     ← waiting on dongle 4
#3  coder_routine (arg=0x555555559130) at codexion.c:141

Every coder is blocked in take_dongle at line 88, each on a different mutex address. That's the signature of circular wait (Course 7): everyone holds one dongle and waits for the next. The backtraces make the cycle plain — you don't have to guess.

2.3  What GDB can't do

ProblemGDB?
Segfault — where and whyYes — backtrace at the crash site
Deadlock — which threads, which locksYes — thread apply all bt
Infinite loop / livelockYes — attach and see where threads are spinning
Data race that hasn't manifestedNo — nothing to break on; the code looks fine
Lock-order inversion that hasn't deadlocked yetNo — no symptom exists to catch
Connect to Codexion GDB is for problems that happened. It has a location, a moment, a stack. The races and latent lock-order bugs the evaluation scale hunts for have none of those — they're properties of your code's structure, not events. That's a different tool entirely, and it's the rest of this course.

Chapter 03

Helgrind

3.1  How it detects races without them happening

The key idea Helgrind doesn't wait for a race to produce wrong output. It builds the happens-before relation (Course 4) as your program runs — tracking which memory accesses are provably ordered relative to each other by locks, thread creation, joins, and condition signals. Then it applies the definition directly: two accesses, same address, at least one a write, from different threads, with no happens-before edge between them = data race. Reported, whether or not it caused any visible problem on this run.

This is why Helgrind finds bugs your testing never will. It's not observing outcomes; it's checking structure.

valgrind --tool=helgrind ./codexion 5 2000 200 200 200 10 0 fifo

3.2  Reading a race report

==12345== Possible data race during write of size 4 at 0x520A0 by thread #3
==12345==    at 0x109332: take_dongle (codexion.c:88)
==12345==    by 0x1094A1: coder_routine (codexion.c:141)
==12345==  Locks held: none
==12345== 
==12345== This conflicts with a previous read of size 4 by thread #2
==12345==    at 0x1092F1: dongle_is_free (codexion.c:71)
==12345==    by 0x109488: coder_routine (codexion.c:139)
==12345==    Locks held: none

Read it top to bottom:

  • Two accesses, same address (0x520A0) — your dongle's flag.
  • Different threads (#3 and #2), at least one a write. The race definition is satisfied so far.
  • Locks held: none — the smoking gun. Had both accesses occurred under the same mutex, Helgrind would see the happens-before edge through the lock's acquire/release and stay silent.
  • Exact lines — codexion.c:88 and codexion.c:71. Go look at line 71: it's almost certainly a check outside the mutex.
"Locks held: none" is the whole message If you learn one thing about reading Helgrind: that line tells you the accesses were unsynchronized. If it instead listed a mutex held by both, there'd be no report. Seeing it means you touched shared mutable state outside the lock — the single most common Codexion bug, usually from an "optimization" that checks availability before bothering to lock.

3.3  Lock-order detection

Helgrind also records the order each thread acquires locks and flags inconsistencies across threads — even if no deadlock occurred:

==12345== Thread #3: lock order "0x5A0 before 0x5B0" violated
==12345== Observed (incorrect) order is: acquisition of lock at 0x5B0
==12345==    at 0x109440: take_dongle (codexion.c:102)
==12345==  followed by a later acquisition of lock at 0x5A0
==12345==    at 0x109440: take_dongle (codexion.c:102)
Connect to Codexion — this is graded This is Helgrind finding your latent circular wait (Course 7) before it ever deadlocks. The scale is unambiguous: "If any of these tools reports a data race or a lock-ordering deadlock, tick the appropriate flag" — meaning 0. Your left-then-right dongle acquisition will produce exactly this report. Fix it with consistent ordering (lowest id first), and the report disappears because the inconsistency is gone.

DRD & ThreadSanitizer

4.1  Three tools, one theory

HelgrindDRDThreadSanitizer
MechanismValgrind binary instrumentationValgrind binary instrumentationCompile-time instrumentation
Invocationvalgrind --tool=helgrind ./xvalgrind --tool=drd ./xcc -fsanitize=thread … then run
Slowdown~20–50×~20–50×~5–15×

Recompile neededNoNoYes

All three implement variations on happens-before analysis. They differ in internal algorithms, heuristics, and memory-state tracking — which is precisely why they don't always agree.

4.2  Why the scale wants all three

From the evaluation scale The scale lists all three explicitly: valgrind --tool=helgrind, valgrind --tool=drd, and "recompiling the project with -fsanitize=thread and running it." That's not redundancy — each tool's heuristics differ enough that one can find what another misses. DRD tracks per-thread state differently from Helgrind; TSan's compile-time instrumentation sees things binary instrumentation can't infer.

Run all three yourself, before your defense. It costs minutes and removes the possibility of being surprised.

# the full sweep
valgrind --tool=helgrind ./codexion 5 2000 200 200 200 10 0 fifo
valgrind --tool=drd      ./codexion 5 2000 200 200 200 10 0 fifo
cc -Wall -Wextra -Werror -pthread -fsanitize=thread -g *.c -o codexion_tsan
./codexion_tsan 5 2000 200 200 200 10 0 fifo

4.3  Reading a TSan report

WARNING: ThreadSanitizer: data race (pid=4242)
  Write of size 4 at 0x7b0400000d20 by thread T3 (mutexes: write M12):
    #0 take_dongle codexion.c:88

  Previous read of size 4 at 0x7b0400000d20 by thread T2:
    #0 dongle_is_free codexion.c:71

  Location is heap block of size 240 at 0x7b0400000d00 allocated by main thread

TSan's output is arguably more readable than Helgrind's. Note (mutexes: write M12) on the first access but nothing on the second — thread T3 held mutex M12 while writing; thread T2 held nothing while reading. That asymmetry is TSan saying the same thing Helgrind's "Locks held: none" says: one side was unsynchronized. TSan also tells you the racy address is inside a heap block allocated by main — that's your dongle array.

Connect to Codexion — timing is meaningless here Under any of these tools, your program runs 5–50× slower and threads are largely serialized. Coders will burn out in configurations that are perfectly feasible natively. That is expected and is not a bug. Use these tools to check races and lock ordering; use a native build to check timing (Course 8, Chapter 5). If an evaluator points at the timestamps under Helgrind, this is your answer.

Chapter 05

Finding races

5.1  The five races you'll actually write

RaceWhy it happensReport
Unprotected availability check — if (d->free) before lockingFeels like an optimization: avoid locking when obviously busy"Locks held: none" on the read
Monitor reading last_compile_start unlocked"It's just a read, and I only need an approximate value"Race between monitor thread and coder
Unprotected stop flag — while (!sim->stop)"It's one int, writes are atomic anyway"Race between monitor (write) and every coder (read)
Unprotected compile counter — coder->compiles++Looks like one statement; it's load-add-store (Course 4)Race, and undercounted totals
Reading heap state outside the lock — heap_peek() to "just check"Peeking feels read-only and harmlessRace with any concurrent insert/pop
The pattern Every single one is the same mistake wearing a different hat: "this particular access is too small/too read-only/too approximate to need the lock." There is no such access. If shared mutable state is involved, the lock is involved. No exceptions, no optimizations, no "just a bool."

5.2  The stop flag deserves its own paragraph

This one is worth dwelling on because the reasoning behind it is so seductive. "It's a single int. The monitor writes 1, coders read it. A torn read is impossible on x86. What could go wrong?"

Two things. First, the compiler may hoist the read out of your loop entirely — nothing in the C memory model tells it another thread can change that variable, so while (!sim->stop) can legally become if (!sim->stop) while (1). Your coder never stops. Second, and this is what actually gets flagged: an unsynchronized concurrent read/write is undefined behavior in C, regardless of what the hardware would have done. Helgrind and TSan will both report it, and the scale says any reported race is a 0.

The fix Read and write the stop flag under the same mutex that protects the rest of your simulation state. It's the natural place anyway — your coders already lock that mutex to check dongle availability, so checking stop in the same critical section is free.

5.3  A workflow that finds them

  1. Run all three tools on the feasible benchmark (5 2000 200 200 200 10 0 fifo). Clean output is the baseline.
  2. Run on the contended case (5 3000 200 200 200 10 800 fifo). More contention exercises more interleavings.
  3. Run the edf variant. Different code path through your heap.
  4. Run the single-coder case (1 800 200 200 200 10 0 fifo). The degenerate path is where the odd bugs live.
  5. For every report: find the two line numbers, identify which access lacks the lock, fix it. Never suppress a report you don't understand.
  6. Re-run until all three are silent on all four configurations.
Connect to Codexion This exact workflow is what the evaluator will run. Running it first turns your defense from a discovery process into a confirmation. Roughly ten minutes of work against the possibility of a 0.

Chapter 06

Finding deadlocks

6.1  Recognizing what you're looking at

Your program stops producing output. Three different things could be happening, and they need different responses:

SymptomCPU usageDiagnosis
No output, program alive~0%Deadlock — everyone blocked in futex_wait
No output, program alive~100%Livelock or infinite loop — threads running, making no progress
Output continues, one coder silentNormalStarvation (Course 7) — not a deadlock at all
Hangs at exit, after the last log line~0%A thread never woke — your pthread_join is waiting on a coder still in cond_wait
Check CPU first top or htop distinguishes deadlock from livelock in two seconds, and they're completely different bugs. 0% means everyone's asleep waiting for each other. 100% means everyone's awake, busy, and achieving nothing.

6.2  The hang-at-exit bug

This is the deadlock most Codexion implementations actually hit, and it's not in the coder logic at all — it's in shutdown:

// main, after the monitor returns
for (int i = 0; i < n; i++)
    pthread_join(tids[i], NULL);   // ← hangs here, forever

Why: the monitor set stop = 1 and called pthread_cond_signal — waking one waiter. The other four coders are still blocked in pthread_cond_wait, will never be woken, and will never return from coder_routine. pthread_join waits for them forever. Your last log line printed correctly; the program just never exits.

The fix pthread_cond_broadcast (Course 6, Chapter 3). One event that every waiter must observe is the definition of a broadcast case. And the coders' wait predicate must include the stop flag: while (!sim->stop && !can_take_dongles(coder)) — otherwise a woken coder re-checks only the dongle condition, finds it false, and goes straight back to sleep despite the simulation being over.

GDB confirms it instantly: thread apply all bt shows four threads in pthread_cond_wait and the main thread in pthread_join. That shape is unmistakable.

6.3  Latent vs. actual

The distinction that matters at defense An actual deadlock hangs the program — you find it with GDB, on a run where it happened. A latent deadlock is a lock-order inversion that hasn't been unlucky yet: it will hang, eventually, on some interleaving. GDB cannot find it, because nothing has gone wrong. Helgrind finds it anyway, by comparing acquisition orders across threads (Chapter 3).
Connect to Codexion This is why "my program has never deadlocked in 200 runs" is not a defense. The scale's instruction to run Helgrind exists precisely to catch latent inversions. A left-then-right dongle acquisition may run cleanly all evening and still be flagged in the first ten seconds of a Helgrind run — because the inconsistent order is in the code, whether or not the timing has exposed it yet.

7.2  Reading your own logs as evidence

The subject's log format is not decoration — it's a diagnostic instrument. The scale tells evaluators exactly what to look for, which tells you exactly what to check first:

  • Every "is compiling" is preceded by exactly two "has taken a dongle" lines for the same coder. One means a bug in your acquisition. Three means you're double-counting.
  • No dongle is taken twice without an intervening release. Track each dongle's id through the log. A violation is dongle duplication — an automatic 0.
  • No "has taken a dongle" for a given dongle within dongle_cooldown ms of its release. This is directly checkable by timestamp arithmetic; the scale's 400ms cooldown case exists for it.
  • Phase durations match the parameters (±tolerance). A compile that takes 340ms when time_to_compile is 200 means you're holding dongles too long.
  • "burned out" is the last line, full stop.
Do this before your defense Run the scale's own benchmark cases and check these five properties by hand — or write a small script to check them. Every one of these is something the evaluator will look for. Finding a violation yourself is a bug fixed; finding it during the defense is a grade lost.

7.3  The pre-defense checklist

# 1. Clean build, no warnings
make re                                   # -Wall -Wextra -Werror -pthread

# 2. The scale's four benchmark cases, native, several runs each
./codexion 1 800  200 200 200 10 0   fifo   # must burn out ~800ms
./codexion 5 2000 200 200 200 10 0   fifo   # no burnout, stops at 10 compiles
./codexion 5 2000 200 200 200 7  0   edf    # same, edf path
./codexion 5 500  200 200 200 10 0   fifo   # must burn out ~500ms, last line

# 3. Contention + cooldown, compare grant order
./codexion 5 3000 200 200 200 10 800 fifo
./codexion 5 3000 200 200 200 10 800 edf

# 4. Race and lock-order sweep (ignore timings here)
valgrind --tool=helgrind ./codexion 5 2000 200 200 200 10 0 fifo
valgrind --tool=drd      ./codexion 5 2000 200 200 200 10 0 fifo
cc -Wall -Wextra -Werror -pthread -fsanitize=thread -g *.c -o cx_tsan && ./cx_tsan 5 2000 200 200 200 10 0 fifo

# 5. Leaks
valgrind --leak-check=full --show-leak-kinds=all ./codexion 5 2000 200 200 200 10 0 fifo

# 6. The recode: can you turn fifo into lifo in a few minutes?
Connect to Codexion — that last line The scale includes a recode: "turn the fifo scheduler into a lifo one." If your ordering policy is properly isolated in your heap's comparison function, that's a two-line change and you'll do it calmly in front of the evaluator. If your fifo logic is smeared across your coder routine, you'll be rewriting under pressure. The scale even says it will check "the change is made in the scheduler/queue logic itself and not at a superficial level" — which is really a test of whether you separated policy from mechanism (Course 6, Chapter 7) in the first place.
