Mutexes

A mutex looks like the simplest thing in concurrent programming: lock, do the work, unlock. Underneath it is a compare-and-swap instruction, a kernel hash table of sleeping threads, a fast path that never enters the kernel at all, and a set of guarantees that go well beyond "one at a time." This course opens all of it — and, critically, shows you exactly what a mutex cannot do, which is why Codexion makes you build a scheduler on top of one.

Chapter 01

Why mutexes exist

1.1  The problem, stated precisely

Why this exists Threads share memory (Course 4). Sharing means two coder threads can execute conflicting read-modify-write sequences on the same dongle state, interleaved arbitrarily by the scheduler, producing lost updates and corrupted structs. You need a way to say: "from here to here, exactly one thread at a time, no exceptions." That region is a critical section, and a mutex (mutual exclusion lock) is the primitive that enforces it.

The contract is simple to state and surprisingly hard to implement correctly: at most one thread may hold the mutex at any instant; a thread attempting to lock a held mutex waits until it's free; unlocking releases it and lets exactly one waiter proceed.

1.2  Why you can't just write it yourself with a flag

The naive attempt fails immediately:

// BROKEN — do not do this
while (dongle->locked) { /* spin */ }
dongle->locked = 1;      // ← two threads can both reach here
/* critical section */
dongle->locked = 0;

The check and the set are two separate operations. Two threads can both read locked == 0, both fall out of the loop, and both set it to 1 — both now believe they hold the lock. This is precisely Course 4's lost-update race, applied to the lock itself. Fixing it requires hardware support: a single instruction that reads and writes atomically, with no window in between (Chapter 2).

Analogy — the single-occupancy room A mutex is a room with one key hanging outside. The rule "check if the key is there, then take it" only works if checking-and-taking is one indivisible motion. If two people can both glance at the hook and both reach for the key, you don't have a lock — you have two people in a one-person room, each certain they're alone.

1.3  Two guarantees, not one

A mutex provides both:

  • Mutual exclusion — the obvious one: one thread at a time in the critical section.
  • Memory ordering (acquire/release, Course 4 Chapter 5) — the less obvious but equally load-bearing one: the next thread to acquire is guaranteed to see everything the previous holder wrote, in order, not a stale or reordered view.
Connect to Codexion The subject's rule — "to prevent coders from duplicating dongles, you must protect each dongle's state with a mutex (pthread_mutex_t)" — is the direct application. "Duplicating a dongle" is exactly the lost-update race: two coders both observe a dongle free, both take it, and now one physical dongle is held twice. The mutex closes that window.

Chapter 02

Internal implementation

2.1  Compare-and-swap: the hardware primitive

Why this exists Chapter 1 showed that "check then set" must be indivisible. CPUs provide exactly this as a single instruction: compare-and-swap (CAS) — atomically compare a memory location to an expected value and, only if they match, write a new value; report whether it succeeded. On x86-64 this is lock cmpxchg; the lock prefix asserts exclusive ownership of the cache line for the duration (Course 4, Chapter 4).
// The correct version of Chapter 1's broken loop:
int expected = 0;
if (atomic_compare_exchange_strong(&lock_word, &expected, 1)) {
    /* we own the lock — no other thread could have slipped in */
}

There is no window between the compare and the swap. Either you observed 0 and wrote 1, atomically, or you didn't — no third outcome exists.

2.2  Spinning vs. blocking

CAS tells you whether you got the lock. It doesn't tell you what to do if you didn't. Two options:

SpinningBlocking
BehaviorRetry the CAS in a tight loop until it succeedsAsk the kernel to put this thread to sleep until woken
Cost when the wait is very shortCheap — no syscall, no context switchWasteful — you pay a syscall + 2 context switches to wait 50ns
Cost when the wait is longTerrible — burns a whole core doing nothing for millisecondsCheap — the core runs other useful threads meanwhile
The insight that produced futexes Neither answer is right for all cases — so the correct design is to do both: spin (or rather, just try the CAS) in userspace for the uncontended case, and only fall into the kernel when there's genuine contention worth sleeping through. That hybrid is exactly what a futex is (Chapter 3).

2.3  Why pthread_mutex_t is 40 bytes

If a mutex were just "one bit: locked or not," sizeof(pthread_mutex_t) would be tiny. On glibc/x86-64 it's 40 bytes, because it also carries:

  • The futex word itself (the atomically-manipulated lock state)
  • The owner's thread ID — needed for error-checking mutexes and recursive mutexes (Chapter 7)
  • A recursion counter (for PTHREAD_MUTEX_RECURSIVE)
  • The mutex kind (normal / error-checking / recursive / adaptive)
  • Priority-inheritance bookkeeping (Chapter 6)
Connect to Codexion This is the honest answer for your README's "thread synchronization mechanisms" section: a pthread_mutex_t guarding a dongle isn't an abstract "lock" — it's a futex word with a userspace CAS fast path, a kernel-mediated slow path, and owner/kind bookkeeping you inherit for free from glibc rather than hand-rolling.

Chapter 03

Futexes

3.1  Fast Userspace muTEX

Why this exists Before futexes, every mutex operation on Linux went through the kernel — even locking a mutex nobody else wanted. That's a syscall (hundreds of nanoseconds, a mode switch) for an operation that, in the uncontended case, needs nothing but one atomic instruction. The futex design flips the default: stay entirely in userspace unless there's actual contention worth involving the kernel for.

Hence the name: fast userspace mutex. The kernel only learns a mutex exists at the moment someone actually has to wait on it.

3.2  The fast path

// simplified pthread_mutex_lock() fast path
int expected = 0;
if (atomic_compare_exchange(&futex_word, &expected, 1))
    return 0;   // acquired — ZERO syscalls, a few nanoseconds
// otherwise: contended, fall through to the slow path

In Codexion with generous timing parameters and low contention, the overwhelming majority of your dongle lock/unlock pairs never enter the kernel at all. This is why "I'm calling a mutex thousands of times, isn't that slow?" has a satisfying answer: no — an uncontended lock is a handful of CPU cycles.

3.3  The slow path: FUTEX_WAIT / FUTEX_WAKE

When the CAS fails, glibc calls the futex() syscall:

futex(&futex_word, FUTEX_WAIT, expected_val, timeout);

Semantics: atomically re-check that futex_word still equals expected_val; if so, sleep on this address. The atomicity of that recheck is essential — it closes the window where the holder might unlock between your failed CAS and your decision to sleep (otherwise you'd sleep forever waiting for a wake that already fired).

On unlock, the holder resets the word and, if it detected waiters, issues:

futex(&futex_word, FUTEX_WAKE, 1);   // wake one waiter
The kernel only tracks this mutex because someone actually had to wait on it. No contention = no kernel involvement at all.

Chapter 04

Lock acquisition

4.1  The full acquisition path

Putting Chapters 2–3 together, a complete pthread_mutex_lock():

  1. Try the CAS. Free → acquired, return. Done, no kernel.
  2. Adaptive spin (for PTHREAD_MUTEX_ADAPTIVE_NP): retry briefly, betting the holder will release within a few hundred cycles — cheaper than sleeping if the bet pays off.
  3. Mark contention. Set the futex word to a value meaning "locked, and waiters exist" so the unlocker knows it must issue a FUTEX_WAKE.
  4. FUTEX_WAIT. Sleep. The task enters TASK_INTERRUPTIBLE, leaves the runqueue, consumes zero CPU.
  5. Woken. Loop back to step 1 — because waking does not mean you own the lock (Chapter 5).
Step 5 is the subtle one Being woken from FUTEX_WAIT means "the lock might be available now," not "the lock is yours." Another thread could grab it between your wake and your retry. This is why the implementation loops rather than assuming success — the same structural reason you must loop around pthread_cond_wait() (Course 6).

4.2  trylock and timedlock

FunctionBehavior on a held mutex
pthread_mutex_lockBlocks indefinitely until acquired
pthread_mutex_trylockReturns EBUSY immediately — never blocks
pthread_mutex_timedlockBlocks until acquired or the absolute deadline passes, then returns ETIMEDOUT
Connect to Codexion — a tempting wrong turn trylock looks like an attractive deadlock avoidance trick: "try both dongles, if the second fails, release the first and retry." It does prevent circular wait — but naively it invites livelock, where all coders repeatedly grab-fail-release in lockstep, making no progress while burning CPU. Note also that pthread_mutex_trylock is not in Codexion's allowed external-function list — the subject steers you toward condition variables for arbitration instead, which is a hint about the intended design, not an arbitrary restriction.

4.3  Who wins when several are waiting?

Critical fact: a plain pthread_mutex_t makes no fairness guarantee whatsoever. POSIX does not promise FIFO ordering among waiters. Whichever thread the kernel happens to wake, or whichever new arrival happens to win the CAS first (a "barging" thread that never waited at all), gets the lock. A thread can, in principle, be starved indefinitely.

Connect to Codexion — this is why you build a scheduler This is the single most important idea in this course for your project. Codexion demands "fair arbitration is mandatory: when multiple coders request the same dongle, the dongle must grant access according to scheduler (fifo or edf)." A raw mutex cannot deliver that — it has no notion of arrival order or deadlines. That's precisely why the subject makes you build an explicit request queue (your heap) on top: the mutex protects the queue's integrity, while your own FIFO/EDF logic decides who gets served next. Mutex = data protection. Your heap = arbitration policy. Two different jobs.

Chapter 05

Wait queues

5.1  The kernel's futex hash table

Why this exists When coder 2 calls FUTEX_WAIT on dongle 3's lock word, the kernel must remember: "this task is sleeping, waiting for that specific address." When someone later calls FUTEX_WAKE on that address, the kernel must find exactly those sleepers. That mapping — address → list of waiting tasks — is the wait queue, implemented as a hash table keyed by the futex address.

Note the elegance: the kernel stores nothing about a futex until someone waits on it. A mutex that's never contended has no kernel-side existence at all — no allocation, no bookkeeping, nothing. The wait queue entry appears on first contention and disappears when the last waiter leaves.

5.2  WAKE one vs. WAKE all

OperationEffectUsed by
FUTEX_WAKE, 1Wake one waiterpthread_mutex_unlock, pthread_cond_signal
FUTEX_WAKE, INT_MAXWake every waiterpthread_cond_broadcast
The thundering herd Waking all waiters when only one can proceed causes a thundering herd: N threads wake, all contend, N−1 immediately go back to sleep, having accomplished nothing but N context switches and a burst of cache-line bouncing. This is why signal is preferred over broadcast when only one waiter can possibly make progress — a distinction that matters directly in Course 6.

5.3  Wait queues you build yourself

The kernel's futex wait queue serves the mutex. It has nothing to do with your queue of coders waiting for a dongle — those are different layers solving different problems:

Kernel futex wait queueYour Codexion request heap
TracksThreads sleeping on a lock wordCoders waiting for a dongle, ordered by policy
OrderingUnspecified / not fair by designStrictly fifo or edf, as the subject demands
Who manages itThe kernel, invisiblyYou, explicitly, in C
Connect to Codexion This table is your answer to "why do I need a heap when mutexes already have wait queues?" — because the kernel's wait queue is unordered by design, invisible to you, and cannot express "the coder with the earliest burnout deadline goes next." Your heap is the arbitration layer; the futex wait queue is just the sleeping mechanism underneath it.

Chapter 06

Priority inheritance

6.1  Priority inversion, the failure mode

Why this exists Consider three threads: L (low priority), M (medium), H (high). L acquires a mutex. H tries to acquire the same mutex and blocks — expected, it must wait for L. Then M (which never touches the mutex at all) becomes runnable and, being higher priority than L, preempts it. Now L can't run, so L can't release the mutex, so H — the highest-priority thread in the system — is blocked indefinitely by M, a thread of lower priority than H. The priority ordering has been inverted.
The Mars Pathfinder mission famously suffered exactly this bug in 1997 — it is not a theoretical concern.

6.2  Priority inheritance as the fix

Priority inheritance: while a low-priority thread holds a mutex that a high-priority thread is waiting for, the holder temporarily inherits the waiter's priority. L runs at H's priority, so M can no longer preempt it; L finishes its critical section quickly, releases the mutex, drops back to its own priority, and H proceeds. The inversion window shrinks to just the length of L's critical section — bounded and acceptable.

pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
pthread_mutex_init(&mtx, &attr);

This is one of the things those 40 bytes of pthread_mutex_t (Chapter 2) pay for: the bookkeeping to track the owner and manage inherited priorities.

6.3  Does Codexion need it?

Honest scope note No. Priority inheritance addresses inversion between threads at different OS scheduling priorities. Codexion's coder threads all run at the same default priority under CFS — you never call pthread_setschedparam or use SCHED_FIFO. With no priority differences, there is no priority inversion to invert. And note: pthread_mutexattr_setprotocol isn't in the subject's allowed function list anyway.
Why learn it then? Two reasons, both practical. First, an evaluator can absolutely ask "what is priority inversion and does your project suffer from it?" — and the strong answer is a precise no, because all my threads share one priority level under CFS, not a blank stare. Second, Codexion's edf mode gives coders different urgencies (deadlines), which is priority-like in spirit — so it's worth being able to articulate clearly why your EDF arbitration is a userspace policy over a dongle, not OS thread priorities, and therefore why priority inheritance is genuinely irrelevant here rather than merely omitted.

Chapter 07

Recursive mutexes

7.1  The mutex kinds

KindRelocking from the same threadUnlocking one you don't own
PTHREAD_MUTEX_NORMALDeadlock — you block forever waiting for yourselfUndefined behavior
PTHREAD_MUTEX_ERRORCHECKReturns EDEADLKReturns EPERM
PTHREAD_MUTEX_RECURSIVESucceeds; increments a counterReturns EPERM
Why NORMAL deadlocks rather than erroring Checking "am I already the owner?" costs a comparison on every single lock — a real cost on the fast path that the vast majority of correct code never benefits from. POSIX's default trades that safety check for speed and declares self-relock undefined. ERRORCHECK exists precisely for when you want the check during development.

7.2  How recursion is tracked

A recursive mutex stores the owner's thread ID and a counter. Locking: if the owner is you, increment and return immediately. Unlocking: decrement; only when the count reaches zero is the mutex actually released and a waiter woken. This means every lock must be matched by an unlock — three locks require three unlocks, or the mutex stays held forever.

pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);
pthread_mutex_init(&mtx, &attr);

7.3  Why recursive mutexes are usually a design smell

The argument against You reach for a recursive mutex when a locked function calls another function that also locks the same mutex — which means you've lost track of which functions assume the lock is held. The recursive mutex lets you stop thinking about it, but the confusion doesn't go away: it just stops producing an immediate deadlock. Worse, it silently breaks the mental model of a critical section — with recursion, "I hold the lock" no longer means "the invariant is intact," because an inner function may have left the shared state half-updated before returning to an outer frame that assumes it's consistent.

The usual better fix: split each function into a public version that locks and calls an internal _locked variant that assumes the caller already holds it. Now lock ownership is explicit in your function names, and the compiler-visible structure matches the actual invariant.
Connect to Codexion Codexion needs no recursive mutexes — and note pthread_mutexattr_settype isn't in the allowed function list, so the question is settled by the subject anyway. Your dongle mutexes each guard exactly one dongle's state; your log mutex guards output. If you find yourself wanting to relock a mutex you already hold, that's a strong signal your locking structure needs simplifying, not that you need a recursive mutex. Being able to say that clearly at defense is worth more than the feature would be.

Common mistakes

8.1  The classic eight

MistakeSymptomHow it's caught
Forgetting to unlock on an early returnDeadlock — everyone else blocks foreverProgram hangs; Helgrind may flag lock-order issues

8.2  The Codexion-specific traps

Trap 1 — sleeping while holding a dongle mutex A coder holds two dongle mutexes to compile, and the compile is simulated with usleep(time_to_compile * 1000). That's correct — the subject explicitly says a coder "must hold two dongles" during that time. But holding a mutex during a long sleep is exactly the pattern that's wrong in most other codebases, so be ready to explain why it's right here: the mutex isn't just protecting a data structure, it's modeling physical possession of the dongle for the duration.
Trap 2 — the partially-protected read Checking if (dongle->available) outside the mutex "just to avoid locking when it's obviously busy" is the single most common Codexion race. Helgrind reports it as a conflicting access with Locks held: none (Course 9). There is no such thing as a safe unprotected read of shared mutable state.
Trap 3 — unprotected logging The subject requires "a displayed state message should not be mixed up with another message" and mandates a mutex around output. Forgetting it produces interleaved log lines under contention — an automatic 0 per the evaluation scale, and one that only appears under load, not in your gentle 5-coder tests.

8.3  A defensive checklist

  • Every lock has exactly one matching unlock on every path out of the function, including error returns.
  • Every access to shared mutable state — read or write — is inside the appropriate mutex.
  • If a thread ever holds two mutexes, all threads acquire them in the same documented global order (Course 7).
  • Every pthread_mutex_init has a matching pthread_mutex_destroy at shutdown, after all threads are joined.
  • Critical sections are as short as the semantics allow — with the deliberate, explainable exception of holding dongles across the compile duration.
  • Run valgrind --tool=helgrind, --tool=drd, and a -fsanitize=thread build. All three clean.
Connect to Codexion That last line isn't optional advice — it's transcribed from the evaluation scale, which instructs evaluators to run exactly those three tools and states that any reported data race or lock-ordering deadlock means 0. Course 9 covers reading their output.
