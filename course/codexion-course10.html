Chapter 01

Architecture

1.1  What the subject forces

Start from the constraints Before designing anything, list what the subject has already decided for you. Good architecture here isn't invention — it's recognizing that the requirements have already narrowed the space to roughly one shape, and building that shape cleanly.
RequirementWhat it forces
"Each coder must be represented by a thread"N coder threads via pthread_create
"A separate monitor thread must detect burnout"One additional thread; N+1 total
"Global variables are forbidden!"All shared state heap-allocated, passed by pointer
"Protect each dongle's state with a mutex"Mutex-guarded dongle state
"Fair arbitration is mandatory… fifo or edf"An explicit request queue — the primitives can't order (Course 5)
"You must implement a priority queue (heap)"Hand-written binary heap; no library
"Logging must be serialized"A dedicated log mutex
"All memory must be properly freed"Every thread joined, every allocation freed, every mutex destroyed

1.2  The shape that falls out

N+1 threads, one heap-allocated shared context, two mutexes with distinct jobs.

1.3  The data model

typedef struct {
    int   id;                    /* 1..n */
    int   compiles;              /* how many completed */
    long  last_compile_start;    /* ms — deadline base for edf AND burnout */
    struct dongle *left, *right;
    struct sim *sim;             /* back-pointer to shared context */
} coder_t;

typedef struct dongle {
    int   id;
    int   held_by;               /* coder id, or 0 if free */
    long  released_at;           /* ms — cooldown base */
} dongle_t;

typedef struct sim {
    /* parsed parameters */
    int   n, t_burnout, t_compile, t_debug, t_refactor, compiles_required, cooldown;
    int   use_edf;

    /* shared mutable state — all guarded by mtx */
    dongle_t *dongles;           /* n dongles */
    coder_t  *coders;            /* n coders */
    heap_t   *queue;             /* pending dongle requests, ordered by policy */
    int       stop;              /* 1 = simulation over */
    long      start_ms;

    pthread_mutex_t mtx;         /* guards everything above */
    pthread_cond_t  cv;          /* signals state changes */
    pthread_mutex_t log_mtx;     /* guards output only */
} sim_t;
Note the one field doing double duty last_compile_start is both the burnout deadline base (+ t_burnout = when this coder dies) and the EDF priority key. The subject defines them with the same expression precisely because they're the same quantity viewed twice. One field, one write site, two readers.
Connect to Codexion Two mutexes, deliberately. mtx guards simulation state; log_mtx guards printf. Separating them means a coder writing a log line doesn't block another coder from checking dongle availability. Merging them would work but needlessly serializes unrelated work — and you'd be holding the simulation lock across an I/O call.

Chapter 02

The scheduler & the heap

2.1  Why a heap at all

The question the subject is really asking Neither primitive you're given can order anything. pthread_mutex_t makes no fairness promise (Course 5). pthread_cond_t wakes an arbitrary waiter (Course 6). So "grant the dongle to the coder with the earliest deadline" cannot come from the primitives — it must come from a structure you build. Every dongle release asks "who's next?", and a binary min-heap answers in O(1) with O(log n) maintenance.

2.2  One comparison function, two policies

The entire difference between fifo and edf should live in one place:

typedef struct {
    int  coder_id;
    long arrival;      /* ms — fifo key */
    long deadline;     /* last_compile_start + t_burnout — edf key */
} request_t;

/* Returns true if a should be served before b. */
static int higher_priority(sim_t *sim, request_t *a, request_t *b) {
    if (sim->use_edf) {
        if (a->deadline != b->deadline) return a->deadline < b->deadline;
        return a->arrival < b->arrival;      /* deterministic tie-break */
    }
    return a->arrival < b->arrival;          /* fifo */
}
The tie-break is required, not optional The subject says: "Due to timestamp precision, equal deadlines may rarely occur… The tie-breaker rule is required to ensure a fully deterministic EDF policy, even in edge cases." Falling back to arrival order is the natural choice — it's already there, and it's total (no two requests share an arrival timestamp and a coder id).

2.3  Why this design survives the recode

Connect to Codexion — the recode is graded The scale asks the evaluator to have you turn fifo into lifo, and explicitly checks "the change is made in the scheduler/queue logic itself and not at a superficial level." With the design above, lifo is: return a->arrival > b->arrival; — one character. That's not luck; it's what "separate policy from mechanism" buys you. If your ordering logic were spread across coder_routine, this would be a rewrite under pressure.

Two implementation notes that matter:

  • Sift depth is O(log n), so recursion is bounded — no stack-overflow risk even at the subject's 200-coder limit. Iterative is fine too.
  • The heap is shared mutable state. Every insert, pop, and even peek happens under sim->mtx. A "harmless" unlocked peek is a data race Helgrind will find (Course 9).

Chapter 03

Condition variables & the wait predicate

3.1  The predicate is the design

The central insight Your condition variable cannot choose who wakes. So don't ask it to. Make "am I the one the heap selected?" part of every coder's wait predicate. Wake everyone, let each check, and only the heap's chosen coder passes. The CV is demoted to a pure sleep/wake mechanism — which is all it was ever capable of being.
static int can_proceed(sim_t *sim, coder_t *c) {
    long now = now_ms(sim);
    return dongle_free(c->left)  && cooled(sim, c->left,  now)
        && dongle_free(c->right) && cooled(sim, c->right, now)
        && heap_peek(sim->queue) == c->id;      /* the heap decides, not the CV */
}

Five conjuncts, each earning its place:

ConjunctDrop it and…
left freeDongle duplication — automatic 0
left cooledCooldown violated; the scale checks this by timestamp arithmetic
right freeSame as left
right cooledSame as left
heap head is meFairness gone — grant order becomes arbitrary, fifo/edf unenforced

3.2  The acquisition

static int acquire_dongles(sim_t *sim, coder_t *c) {
    pthread_mutex_lock(&sim->mtx);
    heap_insert(sim->queue, make_request(sim, c));   /* register, ordered by policy */

    while (!sim->stop && !can_proceed(sim, c))      /* while — never if */
        pthread_cond_wait(&sim->cv, &sim->mtx);

    if (sim->stop) {                                 /* woken to shut down */
        heap_remove(sim->queue, c->id);
        pthread_mutex_unlock(&sim->mtx);
        return 0;
    }

    heap_pop(sim->queue);
    take(c->left, c);   log_state(sim, c, "has taken a dongle");
    take(c->right, c);  log_state(sim, c, "has taken a dongle");
    c->last_compile_start = now_ms(sim);             /* resets burnout AND edf key */
    log_state(sim, c, "is compiling");
    pthread_mutex_unlock(&sim->mtx);
    return 1;
}

Note !sim->stop inside the predicate. Without it, a coder woken by the shutdown broadcast would find can_proceed false and go straight back to sleep — and pthread_join would hang forever (Course 9, Chapter 6). This is the most common Codexion shutdown bug.

3.3  The release

static void release_dongles(sim_t *sim, coder_t *c) {
    pthread_mutex_lock(&sim->mtx);
    long now = now_ms(sim);
    c->left->held_by  = 0;  c->left->released_at  = now;   /* cooldown starts now */
    c->right->held_by = 0;  c->right->released_at = now;
    c->compiles++;
    pthread_cond_broadcast(&sim->cv);    /* state changed — everyone re-checks */
    pthread_mutex_unlock(&sim->mtx);
}
Why broadcast here Different coders wait on different dongles, so a signal might wake a coder whose dongles are still busy while the one that could proceed sleeps on — a lost wakeup (Course 6, Chapter 3). Broadcast is correct. The alternative — one CV per dongle — is more precise and scales better at 200 coders; both are defensible if you can explain the tradeoff.
Connect to Codexion — the cooldown subtlety A released dongle isn't available: cooled() checks now - released_at >= sim->cooldown. But nothing wakes a coder when a cooldown merely expires — no thread runs at that moment. Either the monitor's periodic broadcast covers it, or coders use pthread_cond_timedwait with the cooldown expiry as the deadline. Miss this and coders can sleep past an available dongle. It's the sort of gap the 800ms-cooldown benchmark exists to expose.

Chapter 04

The monitor thread & the stop flag

4.1  Why a separate thread

Why the subject mandates it A burnout must be reported within 10ms of the deadline — but the coder who burns out is, by definition, blocked. It cannot notice its own death; it's asleep waiting for a dongle it will never get. Something else must be awake and watching. Hence: "A separate monitor thread must detect burnout precisely and stop the simulation."

4.2  The implementation

static void *monitor(void *arg) {
    sim_t *sim = arg;
    while (1) {
        pthread_mutex_lock(&sim->mtx);

        /* 1. Everyone finished? */
        if (all_done(sim)) {
            sim->stop = 1;
            pthread_cond_broadcast(&sim->cv);
            pthread_mutex_unlock(&sim->mtx);
            return NULL;
        }

        /* 2. Anyone burned out? */
        long now = now_ms(sim);
        for (int i = 0; i < sim->n; i++) {
            coder_t *c = &sim->coders[i];
            if (now - c->last_compile_start >= sim->t_burnout) {
                sim->stop = 1;                    /* BEFORE printing */
                log_burnout(sim, c, now);         /* the last line */
                pthread_cond_broadcast(&sim->cv); /* wake everyone so join works */
                pthread_mutex_unlock(&sim->mtx);
                return NULL;
            }
        }

        pthread_cond_broadcast(&sim->cv);  /* nudge — covers expired cooldowns */
        pthread_mutex_unlock(&sim->mtx);
        usleep(1000);                       /* sleep OUTSIDE the lock */
    }
}
DetailWhy it's load-bearing
Reads under mtxlast_compile_start is written by coders — an unlocked read is a race Helgrind finds
usleep outside the lockSleeping under it would block every coder for 1ms out of every 1ms
stop = 1 before printingGuarantees burnout is the last line — coders check stop before logging
Broadcast, not signalEvery sleeper must observe stop, or pthread_join hangs
1ms poll1ms granularity + scheduler latency ≪ 10ms budget (Course 8)

4.3  The stop flag protocol

Three rules, and every one is a bug if broken:

  1. Always read and write it under mtx. Not volatile. Not "it's just an int." An unsynchronized concurrent read/write is undefined behavior and a graded race (Course 9, Chapter 5).
  2. Include it in every wait predicate: while (!sim->stop && !can_proceed(...)). Otherwise a woken coder sleeps again and join hangs.
  3. Check it before every log line. That's what makes burnout provably last.
Connect to Codexion — the infeasible case ./codexion 5 500 200 200 200 10 0 fifo: one cycle is 600ms against a 500ms burnout window, so burnout is guaranteed by arithmetic. What's actually graded is the reporting — within 10ms, and the last line. This whole chapter is the answer to that one test case.

Chapter 05

Design decisions

5.1  The decisions worth defending

Every one of these is a real fork in the road. What matters isn't which branch you take — it's that you took it deliberately.

DecisionOptionsRecommended
Mutex granularityOne global sim mutex vs. per-dongle mutexesOne sim mutex. The heap is global state anyway, so per-dongle locks would need the sim lock too — you'd get complexity without concurrency.
Condition variablesOne shared CV vs. one per dongleEither. Shared+broadcast is simpler; per-dongle scales better at 200. Defend the tradeoff.
Cooldown wakeupsMonitor's periodic broadcast vs. cond_timedwaitMonitor broadcast is simpler and already exists. timedwait is tighter.
Deadlock preventionLock ordering vs. atomic both-or-neitherBoth-or-neither — the predicate already checks both dongles under one lock, so hold-and-wait never occurs.
Log mutexSeparate vs. reuse sim mutexSeparate. Don't hold the simulation lock across I/O.
Timestampsgettimeofday vs. clock_gettimegettimeofday — the subject recommends it and it's in the allowed list. Know it's wall-clock (Course 8).

5.2  Which Coffman condition you break

The strong answer Because can_proceed() checks both dongles under a single mutex and a coder takes both or neither, hold-and-wait never occurs — a coder never holds one dongle while waiting for the other. No hold-and-wait means no circular wait means no deadlock, by construction rather than by luck. That's one sentence, and it's exactly what the README's "Blocking cases handled" section needs.

Notice you get this for free. The predicate had to check both dongles anyway (you can't compile with one). Deadlock prevention fell out of a design chosen for other reasons — which is what good design feels like.

5.3  Cleanup

/* after the monitor returns */
for (int i = 0; i < sim->n; i++)
    pthread_join(tids[i], NULL);        /* every coder — unjoined = leak */
pthread_join(monitor_tid, NULL);

pthread_mutex_destroy(&sim->mtx);       /* only after all threads are gone */
pthread_cond_destroy(&sim->cv);
pthread_mutex_destroy(&sim->log_mtx);

heap_free(sim->queue);
free(sim->dongles);
free(sim->coders);
free(sim);
Connect to Codexion The scale flags memory leaks as a 0, and checks with valgrind --leak-check=full. Two subtleties: an unjoined joinable thread leaks its resources (Course 2), and destroying a mutex while a thread still holds or waits on it is undefined behavior. Join first, destroy second, free last — in that order, always.

Chapter 06

Evaluator questions

6.1  The ten you should be able to answer cold

QuestionThe core of a strong answer
"Why threads, not processes?"Coders must share dongle state cheaply. Processes are isolated by design; sharing would need shm/IPC. Threads share by default via CLONE_VM.
"Why are globals forbidden?"A global is implicitly reachable by every thread with no traceability. Heap + explicit pointer makes "who can touch this" visible in the code.
"Why a heap? Mutexes have wait queues."Those queues are unordered by design and invisible to me. POSIX guarantees no fairness. The heap is the only place fifo/edf ordering can exist.
"How do you prevent deadlock?"I eliminate hold-and-wait: both dongles are checked under one mutex and taken together or not at all. Circular wait can't form.
"How do you prevent starvation?"The heap orders every request. Under edf, the nearest deadline is served first, which is provably optimal for feasible task sets.
"Why while, not if?"Three reasons: spurious wakeups, stolen wakeups, and broadcast. A wakeup is a hint, never a promise.
"Why does cond_wait take a mutex?"Unlock and sleep must be atomic, or a signal lands in the gap and is lost — the waiter sleeps forever on an available dongle.
"How is burnout within 10ms?"Monitor polls at 1ms, reads under the lock, sleeps outside it. 1ms + scheduler latency ≪ 10ms.
"How do you know it's thread-safe?"Helgrind, DRD, and TSan all clean on four configurations. Not "it works" — the tools check structure, not outcomes.
"Isn't this Dining Philosophers?"Structurally yes. It adds mandated arbitration (fifo/edf), cooldown, and hard deadlines — a real-time problem, not just a liveness one.

6.2  The traps

"You hold a mutex across a 200ms usleep — isn't that terrible?" No, and here it's required. The subject says a coder "must hold two dongles" for the whole compile. The mutex isn't protecting a data structure for a few instructions — it's modeling physical possession of a dongle. Releasing it mid-compile would let another coder take a dongle that's still plugged in. This deviates from the usual "keep critical sections short" advice deliberately, and I can say why.
"Under Helgrind, coders burn out at wrong times." Expected. Valgrind serializes threads and slows execution 20–50×, so timings are meaningless under it. Helgrind checks races and lock ordering; a native build checks timing. Two different tests, two different builds — never judge one with the other.
"Your edf gives some coders more urgency. Isn't that priority — what about priority inversion?" Different layers. Priority inversion is about OS thread scheduling priorities; all my threads run at the same default priority under CFS. My edf is a userspace arbitration policy over a dongle. No priority differences means no inversion to fix.

6.3  The recode & final checklist

fifo → lifo. The scale asks for it, and checks that the change lives in the queue logic rather than being faked superficially:

/* fifo */  return a->arrival < b->arrival;
/* lifo */  return a->arrival > b->arrival;

One character. Then run ./codexion 5 3000 200 200 200 10 800 fifo before and after and show the grant order reversed in the logs — that's the evidence the scale asks for.

Before you defend
  • make re — zero warnings under -Wall -Wextra -Werror -pthread
  • All four benchmark cases, several runs each, native build
  • Helgrind, DRD, TSan — all clean on all four
  • valgrind --leak-check=full — zero leaks
  • Log audit: two "has taken a dongle" before every "is compiling"; no dongle taken within its cooldown; burnout last
  • README has "Blocking cases handled" and "Thread synchronization mechanisms" — both required by the subject
  • You can do the lifo recode in under two minutes, calmly
