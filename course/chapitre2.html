Chapter 01

pthread_create()

1.1  The API surface

Why this exists Codexion needs 5 (or 200) independent flows of execution, each cycling compile → debug → refactor, all sharing dongle state. pthread_create() is POSIX's answer to "give me a new, independently-scheduled thing that runs this function, inside my existing process." One call, four parameters, and an entire coder thread comes alive.
int pthread_create(pthread_t *thread,
                    const pthread_attr_t *attr,
                    void *(*start_routine)(void *),
                    void *arg);
ParameterRole
threadOutput — filled with an opaque handle identifying the new thread (Chapter 6 covers exactly what this handle is and isn't)
attrOptional attributes: custom stack size, detached vs. joinable, scheduling policy — NULL means "sane defaults"
start_routineThe function this new thread will run — for Codexion, something like void *coder_routine(void *arg)
argA single void* passed to start_routine — this is the thread's only guaranteed private input channel

Return value: 0 on success, a nonzero error code on failure — notably, unlike most POSIX calls, it does not use errno; it returns the error code directly.

1.2  What happens the instant you call it

Underneath that four-parameter call, roughly this sequence runs, entirely inside glibc's NPTL (Native POSIX Thread Library) before the kernel is even involved:

  1. Allocate a new stack region for the thread (via mmap()), sized per attr or a sane default (commonly 8MB, though only pages actually touched become resident, Course 1's demand-paging idea).
  2. Place a guard page at the end of that stack (Chapter 5).
  3. Build an initial machine-state layout on that stack so that when the new thread starts executing, it will effectively call start_routine(arg).
  4. Call the clone() syscall (Chapter 2) with a specific flag combination telling the kernel "create a new schedulable entity that shares my address space, using this stack, starting at this entry point."
  5. The kernel creates the new task, marks it runnable, and pthread_create() returns to the calling thread — often before the new thread has actually run a single instruction; the two threads race from this point on, genuinely concurrent.
Why "returns before the new thread runs" matters There is no guarantee coder thread 3's first instruction executes before coder thread 4 is even created, or before the thread that called pthread_create() five times in a loop finishes that loop. This is precisely why Codexion's "coder N sits between coder N-1 and coder N+1" seating and dongle assignment must be fully decided and passed into each thread's argument struct before any thread starts running — you cannot rely on creation order implying execution order.

1.3  Passing arguments into a new thread

Because arg is a single void*, the standard pattern for handing a new thread more than one piece of data — its coder number, pointers to its left/right dongles, a pointer to shared scheduler state — is to malloc() (or otherwise durably allocate) a small context struct and pass its address:

typedef struct {
    int             id;
    struct dongle   *left, *right;
    struct sched_q  *queue;
} coder_ctx_t;

coder_ctx_t *ctx = malloc(sizeof(*ctx));
ctx->id = i;
ctx->left = &dongles[i];
ctx->right = &dongles[(i + 1) % n];
ctx->queue = shared_queue;
pthread_create(&tids[i], NULL, coder_routine, ctx);
Connect to Codexion — a classic bug A very common mistake: passing &i (the address of a shared loop variable) directly into pthread_create() instead of a per-thread heap-allocated copy. By the time coder thread 3 actually reads *arg, the loop may have already incremented i to 4 or 5 — every coder thread ends up reading the same, final, wrong value. This is exactly the kind of "no global variables" discipline the subject enforces: give each thread its own, independently allocated argument, never a shared loop counter's address.

Chapter 02

clone(): the syscall underneath

2.1  Why Linux has no separate 'thread' syscall

Why this exists Many operating systems textbooks describe "process creation" and "thread creation" as fundamentally different kernel operations. Linux took a different, more general design: there is exactly one syscall for creating any new schedulable flow of execution, clone(), and what makes the result "a thread" versus "a process" is entirely which resources you tell it to share versus copy.

In fact, on Linux, even fork() is implemented as a call to clone() with a particular flag set (mostly nothing shared — separate address space, separate file descriptor table by default, aside from copy-on-write memory pages). pthread_create() calls the exact same underlying syscall, just with a very different flag set.

2.2  The flags that make a thread a thread

clone(fn, child_stack,
      CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND |
      CLONE_THREAD | CLONE_SYSVSEM | CLONE_SETTLS |
      CLONE_PARENT_SETTID | CLONE_CHILD_CLEARTID,
      arg, &parent_tid, tls, &child_tid);
FlagEffect
CLONE_VMShare the entire virtual address space — the single flag that makes the heap shared, and therefore makes this a "thread" rather than an isolated process
CLONE_FSShare filesystem info (cwd, umask)
CLONE_FILESShare the file descriptor table
CLONE_SIGHANDShare signal handler dispositions
CLONE_THREADJoin the same thread group — same PID as reported by getpid()
CLONE_SETTLSInstall a fresh Thread-Local Storage area for the new thread (Chapter 4)
CLONE_CHILD_CLEARTIDOn exit, kernel zeroes a userspace word and futex-wakes it — the actual mechanism pthread_join() waits on

2.3  clone() vs. fork(), side by side

There is no separate kernel concept of "thread creation" — it's clone() either way.
Connect to Codexion This is why Course 1's process-isolation discussion and this course's thread discussion aren't two unrelated topics — they're two settings of the exact same dial. Codexion's mandate to use pthread_create() (rather than, say, forking 5 coder processes) is a mandate to flip that dial toward "share everything except stack/registers/TLS," because that's the resource-sharing profile the dongle-arbitration problem actually needs.

Chapter 03

The Thread Control Block (TCB)

3.1  What a TCB actually holds

Why this exists The scheduler needs to pause a coder thread mid-execution, run four other threads for a while, and later resume that exact same coder thread as if nothing happened — same register values, same stack position, same program counter. Somewhere, the kernel needs a place to park all of that state while the thread isn't actually running. That place is the Thread Control Block (TCB).

Conceptually, a TCB holds:

  • Saved CPU register values (general-purpose registers, stack pointer, program counter, flags) — frozen exactly as they were the instant this thread was last paused
  • Scheduling info: thread state (Ready/Running/Blocked/Terminated — Chapter 6), priority, scheduling class
  • A pointer to the thread's stack
  • A pointer to the thread's TLS area (Chapter 4)
  • Thread identity (Chapter 6) and a pointer back to the owning process's shared resources (address space, file descriptors)

3.2  task_struct: Linux's real TCB

Linux doesn't have a struct literally named "TCB" — the concept is realized by (a subset of the fields inside) task_struct, the same universal schedulable-entity struct Course 1 mentioned exists for both processes and threads. The TCB-relevant fields include:

struct task_struct {
    long              state;       // TASK_RUNNING, TASK_INTERRUPTIBLE, ...
    struct thread_struct thread;   // saved registers, arch-specific
    void             *stack;       // pointer to this task's kernel stack
    struct mm_struct *mm;          // shared address space (same ptr across threads)
    pid_t             pid, tgid;   // thread id, thread-group (process) id
    struct sched_entity se;        // vruntime & scheduling bookkeeping
    /* ... hundreds more fields ... */
};
The key distinction to hold onto thread.regs, stack, pid, and state are unique per thread — this is the "TCB" part. mm is shared by every thread in the same process — this is the part that makes threads threads. One struct, cleanly split between "what's mine alone" and "what I share with my siblings."

3.3  One process, many TCBs

When codexion spawns 5 coder threads and 1 monitor thread, the kernel creates 6 separate task_structs (6 TCBs) — one per thread — all pointing at the same mm_struct, but each with its own independent register state, stack pointer, and scheduling state. ps -eLf or ps -T -p <pid> shows you exactly this: one PID column, six distinct LWP (thread) rows.

Connect to Codexion This is the concrete kernel-level answer to "how can the scheduler run coder 2 while coder 4 is blocked on a mutex, in the same process": each has its own TCB carrying its own independent state field — one can be TASK_RUNNING while another is TASK_INTERRUPTIBLE, simultaneously, because they are, structurally, separate schedulable objects that merely happen to share one address space.

Chapter 04

Thread Local Storage (TLS)

4.1  The problem TLS solves

Why this exists Threads share the heap and globals by design (Chapter 2) — but sometimes you genuinely need a variable that behaves like a global (same name, accessible from anywhere in the thread's call stack, persists across function calls) while still being private per thread. The canonical example: errno. If errno were a true global, one coder thread's failed syscall would stomp on every other coder thread's error-reporting. Thread Local Storage gives you "global-like convenience, per-thread isolation."

4.2  How __thread variables actually work

You can declare your own TLS variables explicitly:

__thread int compiles_done = 0;   // GNU extension
// or, standard C11:
_Thread_local int compiles_done = 0;

Mechanically, on x86-64 Linux: each thread's TCB gets a small private memory block set up at thread-creation time (as part of clone(CLONE_SETTLS, ...)), and the CPU's %fs segment register is loaded with the base address of this thread's block. Every access to a __thread variable compiles down to an address computed relative to %fs, e.g. mov %fs:-0x8, %eax. Same instruction bytes for every thread — but %fs's base differs per thread, so the same instruction resolves to different physical memory depending on who's executing it.

%fs is one of the registers switch_to() reloads on every context switch, so TLS "follows" the running thread automatically.

4.3  errno as the canonical example

glibc's errno is, itself, implemented as a macro that expands to a function call or TLS access returning a per-thread int — not a plain global int errno; like in old single-threaded Unix. This is precisely why every one of Codexion's coder threads can independently call syscalls that might fail (a pthread_mutex_lock misuse, an unlikely gettimeofday error) without any risk of one thread's error code clobbering another's — despite all of them executing the exact same compiled .text.

Connect to Codexion You likely won't declare your own __thread variables in Codexion — heap-allocated per-coder context structs (Chapter 1) cover that need more explicitly and traceably. But understanding TLS is what lets you correctly explain, if asked, why calling a possibly-failing libc function from multiple coder threads simultaneously is safe with respect to errno specifically, even without any mutex protecting it.

Chapter 05

Thread stacks & registers

5.1  Why every thread needs its own stack

Why this exists A stack holds local variables, function call frames, and return addresses — fundamentally, "where was I in this function, and what do I do when it returns." If two coder threads shared one stack, coder 2 calling take_dongle() while coder 4 is simultaneously three functions deep inside its own take_dongle() call would scribble over each other's local variables and return addresses within microseconds. Each thread needs a completely private stack precisely because "where am I in my own call chain" must never be shared, even though the heap deliberately is.

5.2  Guard pages & stack overflow

glibc allocates each pthread's stack via mmap() and places a guard page — mapped PROT_NONE, meaning any access faults immediately — right past the end the stack grows toward. This turns "silent heap/adjacent-stack corruption from an overflowing stack" into "immediate, loud segfault at the exact moment of overflow," which is a far easier bug to diagnose (and, for Codexion, a far more honest failure mode given the subject's zero-tolerance rule for unexpected crashes: better to crash cleanly and immediately at the true cause than corrupt memory and crash confusingly somewhere else later).

Deep or unbounded recursion (e.g. a buggy heap-sift) hits the guard page and faults immediately.

5.3  Registers: what's private, what changes on a switch

Registers are, by hardware definition, singular per core — there's exactly one physical %rax, one %rsp, per core, not one per thread. What makes each thread's register state "private" is that the kernel saves the full register set into that thread's TCB the instant it's paused, and restores a completely different thread's saved values when it resumes. From a coder thread's own perspective, its registers are entirely its own; the underlying hardware is shared and swapped invisibly.

  • General-purpose registers, stack pointer, program counter, flags — saved/restored on every context switch
  • %fs base (TLS pointer, Chapter 4) — reloaded so per-thread variables resolve correctly after the switch
  • Floating-point/vector registers — often saved lazily, only if the outgoing thread actually used them, since not every thread touches floating point
Connect to Codexion This is why "which coder am I" can never be answered by looking at a CPU register directly — registers are a hardware resource borrowed and returned by whichever thread happens to be scheduled onto a core at a given instant. The only reliable way a coder thread knows its own identity is through data explicitly passed to it (the arg pointer from Chapter 1) or through its thread ID (Chapter 6) — never through some register the OS conveniently sets aside "for coder 3 specifically," because no such register exists.

Chapter 06

Thread IDs, lifecycle & states

6.1  pthread_t vs. the kernel TID

Why two different IDs exist POSIX threads are meant to be portable across many operating systems, not all of which implement threads the way Linux does internally. So POSIX defines its own abstract, opaque identifier type, pthread_t, guaranteed only to be comparable via pthread_equal() — not necessarily an integer, not necessarily meaningful outside the pthread library. Linux, separately and specifically, also assigns every thread (every task_struct) a genuine kernel-level integer ID, retrievable via gettid() (a raw syscall, not part of POSIX).
pthread_tKernel TID (gettid())
ScopeMeaningful only within your process, via the pthread libraryA real, globally unique-at-a-point-in-time kernel integer, visible to tools like ps -T
PortabilityPOSIX-standard, portable across POSIX-compliant systemsLinux-specific
Typical Codexion usePassed to pthread_join() to wait for a specific coder thread to finishEssentially never needed directly — your own logging already tracks "coder N" explicitly

6.2  Birth to death: the thread lifecycle

A coder thread's life, end to end:

  1. Creation — pthread_create() returns a pthread_t; the thread is now Ready/Running, executing coder_routine.
  2. Execution — cycles through Running and Blocked states (Chapter 6.3) as it compiles, waits for dongles, debugs, and refactors.
  3. Termination — the thread function returns, or the thread calls pthread_exit() explicitly. Either way, the thread is now "terminated" but its resources (exit status, stack) aren't necessarily reclaimed yet.
  4. Joining (or detaching) — another thread calls pthread_join(tid, NULL), which blocks until that specific thread terminates and then reclaims its resources; alternatively, a thread created "detached" has its resources reclaimed automatically on exit, with no other thread able to join it.
Why joining matters even for a program that "runs until burnout" A thread that terminates but is never joined (and wasn't created detached) leaks its kernel-held resources for the life of the process — a "zombie" thread, conceptually parallel to a zombie process. Codexion's monitor thread needs to cleanly signal all coder threads to stop (on burnout or on reaching number_of_compiles_required) and then pthread_join() every one of them before the program exits, both for a clean shutdown and to avoid the memory-leak class of bug the subject explicitly grades on.

6.3  Ready, Running, Blocked, Terminated

Scheduler moves Ready→Running; a blocking call moves Running→Blocked; wakeup moves Blocked→Ready; return/pthread_exit moves Running→Terminated.
StateCodexion moment
ReadyCoder has both dongles and is about to compile, but hasn't been given a core yet
RunningActually executing on a core — compiling, or running the C code that logs "is debugging"/"is refactoring" and then sleeps
BlockedWaiting inside pthread_mutex_lock() for a dongle, or inside pthread_cond_wait()/pthread_cond_timedwait() for availability to change
TerminatedCoder thread's routine has returned (program is stopping — burnout detected, or every coder hit number_of_compiles_required) and awaits pthread_join()
Connect to Codexion Every line in your log output is a snapshot of one thread's transition through exactly this state machine, filtered through Codexion's own domain vocabulary. "Has taken a dongle" is the moment a Blocked coder becomes Ready/Running again. "Burned out" is the monitor thread detecting a coder that should have transitioned back to Running (via re-acquiring dongles) but didn't, within the allotted time — a liveness failure expressed in exactly the state-machine terms this chapter describes.
