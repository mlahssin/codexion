How programs become running threads

Before touching mutexes or schedulers, you need the ground floor: what your .c files actually turn into, what an executable file really contains, what happens the instant you run it, and why the OS gives you processes and threads instead of one flat pile of shared memory. This course walks the whole pipeline — source code → ELF → process → address space → thread — so that every later idea about compiling, debugging, and refactoring coders has solid ground underneath it.

Chapter 01

From source code to executable

1.1  Preprocessing

Why this exists Before gcc/cc can understand your C code as C code, it needs your file to contain nothing but pure C — no #includes pointing at other files, no #define macros still unexpanded, no conditional #ifdef blocks left undecided. The preprocessor is a dumb, mechanical text-substitution pass that resolves all of that first, so the real compiler only ever has to deal with one self-contained blob of C.

Run it yourself on a Codexion source file and you'll see exactly what the compiler sees:

cc -E codexion.c -o codexion.i

Every #include <pthread.h> is replaced, literally, by the entire contents of pthread.h (recursively — pthread.h itself includes other headers). Every macro is expanded. The result, codexion.i, is often tens of thousands of lines for a file that started at a few hundred — almost all of it declarations pulled in from system headers your code needs but didn't write.

1.2  Compiling to assembly

The compiler proper takes that preprocessed C and translates it into assembly — human-readable-ish mnemonics for the target CPU's instruction set (x86-64 mov, call, lock cmpxchg, etc.), plus symbolic labels for functions and jumps that haven't been resolved to final addresses yet.

cc -S codexion.i -o codexion.s

This is the stage where high-level constructs disappear into their machine-level reality. A call to pthread_mutex_lock(&dongle->mtx) becomes a sequence of instructions loading the address into a register and issuing a call instruction to a symbol named something like pthread_mutex_lock@PLT — the actual atomic CAS logic (Volume 2, Chapter 4) lives inside glibc's own compiled code, not yours.

Why keep this as a separate stage Separating "understand the language" (compiling) from "encode as machine bytes" (assembling, next) lets the same compiler front-end target wildly different CPU architectures just by swapping which assembly dialect it emits — and lets you, the programmer, inspect exactly what your C turned into before it becomes opaque binary.

1.3  Assembling to object code

The assembler translates human-readable assembly into actual machine code bytes, bundled into an object file (.o) — but critically, not yet a runnable program. An object file contains machine code with placeholder addresses for anything it doesn't yet know the final location of: calls to functions defined in other files (like pthread_create, which lives in libpthread/glibc, not your file), and references to global data.

cc -c codexion.s -o codexion.o
# equivalently, straight from source:
cc -c codexion.c -o codexion.o

An object file also carries a symbol table — a list of names it defines (e.g. coder_routine) and names it needs from elsewhere (e.g. pthread_mutex_lock) — which is exactly what the next stage consumes.

1.4  Linking

Why this exists Your Codexion program is never really "just your code." It's your .o files plus code from the C standard library (printf, malloc, atoi) plus code from the pthread library (pthread_create, pthread_mutex_lock). Each of those lives in a separately compiled object file or library. The linker's entire job is to stitch all of these together into one coherent executable, resolving every "I need this symbol from somewhere else" reference into a real, final memory address.
cc codexion.o -o codexion -pthread
# Makefile's $(NAME) rule is exactly this final linking step

Two flavors of linking matter here:

  • Static linking — the library's actual machine code is copied directly into your final executable. Bigger binary, no runtime dependency on that library being present on the target machine.
  • Dynamic linking — your executable just records "I need libpthread.so at runtime," and the actual code is loaded and connected when you run the program (by the dynamic linker/loader, ld.so). Smaller binary, but requires the shared library to be installed wherever you run it.
Connect to Codexion This is exactly why your Makefile's compile flags matter: -Wall -Wextra -Werror -pthread — that trailing -pthread isn't decoration. It tells both the compiler (define certain preprocessor macros, use thread-safe versions of some library functions) and the linker (link against the pthread implementation) that this program genuinely uses threads. Forget it, and functions like pthread_create may fail to link at all, or worse, silently link against non-thread-safe library internals on some systems.

Chapter 02

ELF files

2.1  Why a file format at all

Why this exists A finished executable isn't just "a pile of machine code bytes." The kernel needs to know, before running a single instruction: where does the code start? Which bytes are code versus data? What other libraries does this need? Where should each piece be placed in memory? Answering these questions requires a structured file format with a defined header and layout — on Linux, that format is ELF (Executable and Linkable Format).

ELF isn't only used for final executables — it's the same format for object files (.o), shared libraries (.so), and core dumps. That uniformity is deliberate: the same tools (linkers, loaders, debuggers) can parse all of them with one format's worth of logic.

2.2  ELF header & sections

Every ELF file opens with a fixed-size ELF header — you can inspect it directly:

readelf -h codexion
# or:
file codexion

Key fields: a magic number (0x7f 'E' 'L' 'F') identifying the format, whether it's 32/64-bit, the target architecture (x86-64), the entry point address (where execution should first jump to — not necessarily main() directly; it's actually a small C runtime startup routine, _start, that sets up the environment and eventually calls your main), and pointers to two different tables describing the rest of the file: the section header table and the program header table.

Sections (.text, .data, .bss, .symtab, .rodata, and many more) are the linker's view of the file — fine-grained, named regions used while linking and for debugging (objdump -h codexion lists them). They matter for building the binary, but the kernel loading it to run doesn't consult them directly.

2.3  Segments: what the loader actually reads

Why segments are different from sections The kernel, at exec time, doesn't care about the linker's fine-grained bookkeeping (dozens of named sections) — it cares about a much coarser question: "which byte ranges of this file get mapped into memory, at what address, with what permissions (readable/writable/executable)?" That coarser view is the program header table, describing segments — each segment is a contiguous chunk to mmap directly into the new process's address space.
readelf -l codexion
Segment typeWhat it becomes at runtime
PT_LOAD (read+execute)Your compiled code — becomes the .text region, mapped read-only+executable
PT_LOAD (read+write)Initialized globals/statics — becomes .data, mapped read-write
PT_DYNAMICInfo the dynamic linker needs: which shared libraries to load (e.g. libpthread.so.0, libc.so.6) and how to resolve symbols against them
PT_INTERPThe path to the dynamic linker itself (typically /lib64/ld-linux-x86-64.so.2) — the kernel loads this first, and it in turn loads everything your program depends on
Connect to Codexion This is the concrete mechanism behind "linking against pthread": your PT_DYNAMIC segment records that codexion needs the pthread implementation at runtime; PT_INTERP points at the dynamic linker that will actually go find and map it in before your main() ever runs a single instruction. If that shared library isn't present on the grading machine, you get a runtime loader error — not a compile error — which is exactly why the subject cares about a clean, standard build environment.

Chapter 03

Process creation & execve()

3.1  fork() then exec()

Why two separate syscalls You might expect "run this program" to be one syscall: "create a process and load this file into it." Unix instead splits this into two orthogonal operations — fork() (duplicate the current process) and execve() (replace this process's image with a different program) — because that split is dramatically more flexible than a single combined "spawn" call.

When you type ./codexion 5 2000 200 200 200 10 0 fifo in your shell:

  1. The shell calls fork(). The kernel creates a near-identical copy of the shell process — same memory contents (via copy-on-write, so no wasteful full duplication actually happens), same open file descriptors, same everything — but now as a distinct process with its own PID.
  2. In the child copy, the shell calls execve("./codexion", argv, envp).
  3. The parent (the original shell) keeps running, typically calling waitpid() to block until the child finishes — this is how your shell knows when your program has exited and prints a new prompt.
Why the split is powerful Because fork() and execve() are separate, the child process can do arbitrary setup between them, before the new program image is loaded — redirecting stdin/stdout to files or pipes (>, | in your shell), changing its working directory, adjusting resource limits — all using the shell's own familiar, already-running code, before handing control to a completely different program. A single combined "spawn" syscall would need to expose parameters for every one of these possible setup steps directly; the Unix split needs none of that.

3.2  What execve() actually replaces

execve(path, argv, envp) is dramatic: it doesn't create anything new — it replaces the calling process's entire program image in place, while keeping the same PID and most process-level bookkeeping (open file descriptors, unless marked close-on-exec; process ID; parent-child relationships). Concretely, the kernel:

  1. Opens and validates the target file (checking the ELF magic number, checking execute permission).
  2. Discards the calling process's old address space entirely — every existing mapping, all old heap/stack contents, gone.
  3. Reads the new binary's program headers (Chapter 2) and mmaps each PT_LOAD segment at its specified address, creating a brand new address space matching the new program.
  4. If PT_INTERP is present, loads the dynamic linker too, which will later resolve the new program's shared library dependencies before main() runs.
  5. Sets the CPU's instruction pointer to the new binary's entry point (not main() directly — the C runtime startup code first, which sets up argc/argv/environment and calls constructors before finally calling your main).

This is why "the same PID runs a completely different program" is possible and completely ordinary — execve()'s entire purpose is to swap out everything about what code is running while preserving everything about which process is running it.

3.3  Building the initial stack

Part of what execve() sets up is the brand new process's very first stack frame — and this is precisely where your command-line arguments and environment variables get placed, in a layout main(int argc, char **argv, char **envp) is written to expect:

The kernel builds this layout before jumping to the new binary's entry point — argc/argv arrive "for free."
Connect to Codexion This is the literal, final answer to "how do my 8 command-line arguments become argv[1] through argv[8] inside main()": the kernel, during execve(), copies your typed argument strings onto this newly built stack and sets up the pointer array, before your first line of codexion.c — not even your C runtime's startup code — has executed. Your atoi(argv[1]) call for number_of_coders is reading memory the kernel placed there specifically for you.

Chapter 04

Virtual memory

4.1  The MMU & page tables

Why this exists If every process addressed physical RAM directly, two problems appear immediately: processes could read/corrupt each other's memory (no isolation), and a program would need to know, at compile time, exactly which physical addresses happen to be free on whatever machine it eventually runs on (no portability). Virtual memory solves both by inserting a translation layer between every address your code uses and the physical RAM it actually touches.

The Memory Management Unit (MMU), a piece of CPU hardware, translates every virtual address your process generates into a physical address, consulting page tables the kernel maintains per-process. Memory is managed in fixed-size chunks called pages (commonly 4KB on x86-64); a page table is essentially a lookup structure mapping virtual page numbers to physical page frame numbers, plus permission bits (readable? writable? executable?).

4.2  Why every address is a lie

Every pointer value your Codexion code ever sees — the address malloc() returns for your dongle array, the address of a local variable on a coder thread's stack, a function pointer to coder_routine — is a virtual address. It has no fixed relationship to any actual RAM chip location; that relationship is entirely mediated by the current process's page tables, which the kernel can and does change over the process's lifetime (e.g. when new pages are mapped in by malloc/mmap, or swapped to disk and back).

Analogy — the postal redirect Virtual memory is like a mail redirect service: every letter you send to "123 Main St" (a virtual address) actually gets forwarded, invisibly, to wherever the recipient currently lives (a physical address) — and that forwarding rule is entirely private to you; your neighbor's letters to "123 Main St" get redirected somewhere completely different, because their redirect table (page table) is separate from yours.

Two immediate consequences fall out of this for Codexion:

  • Isolation — coder thread code in one codexion process instance can never accidentally address another running instance's memory; there's no virtual address that could even resolve there.
  • Predictability — every run of codexion can assume a similar clean address space layout regardless of what else is running on the machine, because the OS/loader constructs a fresh mapping every time (subject to ASLR randomization for security, which shifts the base addresses but preserves the relative layout).

4.3  Paging & lazy allocation

When you call malloc(sizeof(struct dongle) * number_of_coders), the kernel very often doesn't immediately hand you real, backed physical memory. Underneath glibc's allocator, a large enough request typically becomes an mmap() call, which reserves a range of virtual addresses and marks them in your page table — but the corresponding physical pages may not be allocated until you actually write to them for the first time (a page fault triggers the kernel to find a free physical frame and update the page table on demand). This is demand paging, and it's why allocating a huge buffer you never touch is nearly free, while allocating a smaller buffer you immediately fill triggers real physical memory pressure.

Connect to Codexion This is one honest reason a first access to your freshly malloc'd dongle array can occasionally show up as very slightly slower than subsequent accesses in fine-grained timing measurements — the very first touch of each page may cost a page fault, a detail worth knowing if you ever see a small, consistent timing anomaly only on the first compile cycle and want to explain it accurately rather than blaming your synchronization logic.

Chapter 05

Process memory layout

5.1  .text, .data, .bss

Why this exists A process's address space isn't a flat undifferentiated pile — it's carved into regions because code and different kinds of data have genuinely different lifetimes, mutability, and sharing requirements, and treating them uniformly would throw away real opportunities (like marking code read-only and shareable across processes running the same binary).
RegionComes from which ELF segmentProperties
.textPT_LOAD (read+execute)Your compiled instructions; read-only, executable; can be shared in physical memory across multiple processes running the same binary
.dataPT_LOAD (read+write)Explicitly initialized globals/statics (e.g. int x = 5; at file scope)
.bssSame PT_LOAD, but occupies no space in the fileZero-initialized globals/statics (e.g. int y; at file scope) — the loader just zero-fills this region rather than reading it from disk, saving file space

5.2  Heap vs. stack

Neither the heap nor the (initial) stack corresponds to a segment described directly in the ELF file — they're set up dynamically at runtime, not loaded from disk content:

  • Heap — grows on demand as your program calls malloc() (via the brk() syscall for small/traditional allocations, or mmap() for larger ones). Shared by every thread in the process (Volume 2 covers exactly why).
  • Stack — set up by the kernel at execve() time for the initial thread (Chapter 3); each additional thread created later via pthread_create() gets its own separate stack, privately mapped, not part of the original ELF-derived layout at all.
A typical Linux process address space, low to high addresses. Thread stacks for pthread_create'd threads live in the mmap region, not next to the main stack.

5.3  Mapping ELF segments to runtime regions

Putting the whole pipeline together: your codexion.c compiles down to machine code and initialized/uninitialized global data (Chapter 1); the linker packages that into ELF sections, then groups them into PT_LOAD segments (Chapter 2); at execve() time, the kernel maps each segment at its specified virtual address, becoming your process's .text/.data/.bss regions (Chapter 3); the heap and per-thread stacks are then built up dynamically as your program runs and calls malloc()/pthread_create().

Connect to Codexion This full pipeline is the precise, mechanical reason "global variables are forbidden" bites exactly where it does: anything you declare at file scope ends up baked into .data/.bss, mapped once at process start, implicitly reachable from every thread with zero traceability about who's allowed to touch it when. Your dongle array, by contrast, deliberately lives on the heap — built at runtime, reached only via an explicit pointer you pass into each thread — which is a structural, not just stylistic, difference in how the two kinds of memory come to exist.

Chapter 06

Why processes exist, why threads were invented

6.1  Isolation as the reason for processes

Why this exists Everything in Chapters 1–5 — ELF's careful segment layout, the MMU's address translation, execve()'s full address-space replacement — exists in service of one goal: letting many independent programs run on one machine without any of them being able to corrupt or spy on each other by accident or malice. That goal is isolation, and the process is the unit isolation is built around.

A process gets: its own virtual address space (Chapter 4), its own file descriptor table, its own set of signal handlers, its own resource limits. None of this is shared with any other process by default. This is exactly why a segfault from a stray pointer bug in your codexion binary can never corrupt your terminal, your text editor, or the operating system itself — there is no virtual address in your process's page tables that could even resolve into their memory.

6.2  The sharing problem processes create

Isolation is a feature until you need two things to cooperate closely. Say you wanted 5 independent processes to act as Codexion's 5 coders, all fighting over a shared pool of dongles. Because processes are isolated by design, they cannot see each other's heap-allocated structs at all — coordinating "dongle #2 is now taken" between two separate address spaces requires deliberately opting back into some form of shared state via heavyweight inter-process communication: a shm_open()/mmap() shared memory segment, named semaphores, or message-passing over pipes/sockets — all considerably more machinery than a single malloc()'d struct and a mutex.

Analogy — separate buildings vs. one office Five separate processes are five separate office buildings: secure, but coordinating requires phone calls, couriers, or a shared mailbox system you have to build. Five threads in one process are five desks in one shared office: everyone can just reach across and grab the same physical folder off the same shared shelf (the heap) — fast, but now you need office etiquette (mutexes) so two people don't grab the same folder at once.

6.3  Threads as the fix

Why this exists Threads were invented to give you concurrency — multiple independent flows of execution making progress — without paying the isolation tax when you don't want it. A thread is, at the OS level (concretely: at the clone() level, covered fully in Volume 2), a schedulable unit that deliberately opts back into sharing the parent's address space, file descriptors, and signal handlers, while still getting its own private stack, registers, and scheduling state.

This is precisely the tradeoff Codexion is built to teach: the coders need to share a small, contended pool of dongles cheaply and directly — a textbook case for threads, not processes. Every mutex, condition variable, and heap-based scheduler you build in this project exists because threads chose "share by default, synchronize deliberately" instead of processes' "isolated by default, share only with deliberate machinery."

Connect to Codexion This is the direct answer to "why does the subject mandate threads (pthread_create) instead of letting you fork 5 separate coder processes": five isolated processes would need cross-process IPC just to agree on dongle state, while five threads can share one malloc'd array and coordinate with nothing heavier than pthread_mutex_t/pthread_cond_t — the entire mandatory toolset the subject gives you only makes sense in a shared-memory world.
