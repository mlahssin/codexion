Course 6 of the Codexion series

Condition variables

A mutex tells you when you may touch shared data. It cannot tell you when that data reaches the state you need. Condition variables fill that gap — and they do it with a contract subtle enough that most bugs in multithreaded code live here: the atomic unlock-and-sleep, the wakeups that arrive with no signal behind them, the signals that vanish because nobody was listening yet. This course takes the contract apart completely, then shows exactly where condition variables fit in Codexion's design — and, just as importantly, what they cannot do for you.

Chapter 01

Waiting & sleeping

1.1  The gap mutexes leave

Why this exists A mutex answers "may I touch this data?" It does not answer "has this data reached the state I need?" A coder thread doesn't want the dongle mutex for its own sake — it wants the dongle to be free. If it locks the mutex and finds the dongle busy, it must release the mutex (or nobody could ever free the dongle) and try again later. Mutexes alone give you no way to wait for a condition.

That's the entire gap condition variables fill: a way to say "sleep until someone tells me this state might have changed," without holding the lock that would prevent the change.

1.2  The naive fix: polling

// Works, but wasteful
while (1) {
    pthread_mutex_lock(&mtx);
    if (both_dongles_free(coder)) { take_them(); pthread_mutex_unlock(&mtx); break; }
    pthread_mutex_unlock(&mtx);
    usleep(1000);   // ← guessing
}
Poll intervalProblem
Too short (e.g. 10µs)Burns CPU on constant lock/unlock cycles; hammers the cache line (Course 4)
Too long (e.g. 50ms)Adds up to 50ms of latency after the dongle actually frees — in Codexion, potentially the difference between compiling and burning out
The core problem with polling You're guessing. The information you need — "the dongle just became free" — is known precisely, at the exact instant it happens, by the thread that freed it. Polling throws that knowledge away and reconstructs it by repeatedly asking. A condition variable lets the thread that knows tell the threads that care, with no guessing and no latency.

1.3  Sleeping properly

Recall the thread states (Course 2): a polling thread cycles Ready → Running → Ready, endlessly competing for CPU it can't use productively. A thread blocked on a condition variable enters Blocked — off the runqueue entirely, consuming zero scheduler attention, until an actual event moves it back to Ready. Underneath, this is the same futex sleep mechanism as a contended mutex (Course 5, Chapter 3).

Connect to Codexion This is why the subject offers pthread_cond_t in the mandatory function list and says a condition variable "may be used to manage waiting queues." With 200 coders and tight cooldowns, 199 polling threads would be a genuine performance disaster and would add unpredictable latency to dongle handoffs — which directly threatens the 10ms burnout-detection precision requirement.

Chapter 02

pthread_cond_wait()

2.1  The signature and the contract

int pthread_cond_wait(pthread_cond_t *cond, pthread_mutex_t *mutex);

The contract, stated exactly:

  1. Precondition: the calling thread must already hold mutex.
  2. Atomically: release mutex and begin waiting on cond. No observable gap between the two.
  3. Block until signaled (or spuriously woken — Chapter 5).
  4. Before returning: reacquire mutex. This may itself block if another thread holds it.
Why it takes the mutex as a parameter This surprises people: why does a "wait for a condition" function need a lock? Because steps 2 and 3 must be indivisible. The condition you're waiting on is shared state protected by that mutex — you checked it while holding the lock, and you must not release the lock and go to sleep as two separate steps, or a signal can slip into the gap. Chapter 4 dissects that failure precisely.

2.2  The canonical pattern

pthread_mutex_lock(&dongle_mtx);
while (!both_dongles_free(coder))          // while — never if
    pthread_cond_wait(&dongle_cv, &dongle_mtx);
take_both_dongles(coder);                   // mutex is held again here
pthread_mutex_unlock(&dongle_mtx);

Read it carefully. The mutex is held when you check the condition. It's released while you sleep. It's held again when pthread_cond_wait returns, so the while re-check and the subsequent take_both_dongles() are both safely inside the critical section. At no point do you inspect or mutate shared state without the lock.

2.3  The timed variant

struct timespec deadline;
clock_gettime(CLOCK_REALTIME, &deadline);
deadline.tv_sec += 1;
int rc = pthread_cond_timedwait(&cv, &mtx, &deadline);
if (rc == ETIMEDOUT) { /* the deadline passed */ }

Two things trip people up. First, the timeout is an absolute point in time, not a duration — you must compute "now plus N" yourself. Second, on timeout it still reacquires the mutex before returning; ETIMEDOUT doesn't mean "returned without the lock."

Connect to Codexion pthread_cond_timedwait is in the subject's allowed list, which is a strong hint: it's the natural tool for a monitor thread that must wake to check burnout deadlines within 10ms, and for coders that need to bound how long they'll wait for a dongle whose cooldown expires at a known time. It lets you sleep exactly as long as needed, rather than polling (Chapter 1) or sleeping indefinitely.

Chapter 03

Wakeups: signal vs. broadcast

3.1  The two wake operations

pthread_cond_signalpthread_cond_broadcast
WakesAt least one waiter (unspecified which)Every waiter
Underlying futex opFUTEX_WAKE, 1FUTEX_WAKE, INT_MAX
Use whenExactly one waiter can possibly proceed, and any of them will doThe state change could satisfy several waiters, or different waiters wait on different predicates
"Unspecified which" is not a footnote pthread_cond_signal gives you zero control over which waiter wakes. Not the longest-waiting one. Not the most urgent one. Whichever the kernel picks. If your program cares about the answer — and Codexion's fifo/edf requirement means yours does — the condition variable cannot supply it. Chapter 8 explains what to do instead.

3.2  The thundering herd

Broadcast wakes everyone. If only one waiter can actually proceed, the other N−1 wake up, reacquire the mutex one at a time, re-check their predicate, find it false, and go back to sleep. You've paid N context switches and N mutex acquisitions to accomplish one handoff. With 200 Codexion coders, that's a real, measurable cost — and it lands precisely when contention is highest.

Broadcast is correct here, but wasteful — signal would be cheaper if any single waiter would do.

3.3  When broadcast is actually required

Signal is not always a safe optimization. Use broadcast when:

  • Different waiters wait on different predicates. If coders waiting for dongle 3 and dongle 7 share one condition variable, signalling one waiter might wake a coder whose dongle is still busy — it re-checks, sleeps again, and the coder who could have proceeded never woke. The wakeup was consumed by the wrong thread. This is a genuine lost-wakeup bug, and broadcast fixes it (as does using one condition variable per dongle).
  • One event enables multiple waiters. A "simulation is stopping" flag must wake every coder, not one.
Connect to Codexion — the stop flag When your monitor thread detects burnout, every coder thread blocked in pthread_cond_wait must wake up and notice the simulation is over, or they'll sleep forever and your pthread_join loop will hang. That's a textbook pthread_cond_broadcast — one event, all waiters must observe it. Course 10 covers the full stop-flag design.

Chapter 04

Lost wakeups & why unlock+sleep is atomic

4.1  The race, dissected

The central idea of this course Suppose pthread_cond_wait were implemented naively as two separate steps: unlock the mutex, then sleep. Watch what happens when the timing is unkind.
The lost wakeup. Note the cruelty: the resource IS available, and the waiter never learns.

Signals are not queued. A pthread_cond_signal with no waiters currently registered is simply discarded — it doesn't linger for the next thread to wait. That's not a flaw; it's the definition. A condition variable is stateless: it wakes whoever is waiting right now, and remembers nothing.

4.2  How atomicity closes the window

Now the real implementation. pthread_cond_wait releases the mutex and registers as a waiter atomically — meaning no other thread holding that mutex can observe an intermediate state. Replay the timeline:

  • t1 — Coder 2 holds the mutex, checks, finds the dongle busy.
  • t2 — Coder 2 calls pthread_cond_wait. It becomes a registered waiter and releases the mutex, as one step.
  • t3 — Only now can Coder 5 acquire the mutex. It frees the dongle and signals. Coder 2 is already registered, so the signal finds it.
Why the mutex is the key Coder 5 cannot free the dongle without holding the mutex. Coder 2 checked the condition while holding that same mutex. So Coder 5 is locked out for the entire interval between Coder 2's check and Coder 2's registration as a waiter. There is no window. That's the whole trick — and it's why pthread_cond_wait takes the mutex as a parameter rather than you unlocking it yourself beforehand.

4.3  Underneath: the futex connection

This mirrors FUTEX_WAIT exactly (Course 5, Chapter 3): the kernel atomically rechecks the expected value and sleeps in one syscall, closing the same class of gap. Condition variables and mutexes are built on the same primitive because they face the same fundamental problem: deciding to sleep, based on shared state, must not be interruptible between the decision and the sleep.

Connect to Codexion This is a top-tier defense question. "Why does pthread_cond_wait take a mutex?" has a wrong-sounding-but-common answer ("to protect the condition variable") and a right one: because unlocking and sleeping must be atomic, or a signal can be lost in the gap and a coder sleeps forever on an available dongle. If you can draw the t1–t4 timeline above from memory, you've answered it completely.

Chapter 05

Spurious wakeups

5.1  What POSIX actually permits

Why this exists POSIX explicitly allows pthread_cond_wait to return when nobody signalled. This is a spurious wakeup. It is not a bug in glibc, not a bug in your code, and not something to be fixed — it is documented, legal behavior that your code must tolerate.

Why permit something so apparently user-hostile? Because forbidding it would cost real performance for a guarantee that's free to work around. Guaranteeing "never wake without a signal" requires extra bookkeeping to precisely track which wake corresponds to which waiter — including handling interactions with signal delivery (a POSIX signal interrupting a futex wait causes a return), and glibc's own internal retry loops around futex requeue operations. Every implementation would pay that cost on every wait. Meanwhile, the workaround costs you exactly one keyword.

5.2  while, not if

// WRONG — assumes waking means the condition is true
pthread_mutex_lock(&mtx);
if (!both_dongles_free(coder))
    pthread_cond_wait(&cv, &mtx);
take_both_dongles(coder);        // ← may run when dongles are NOT free!
pthread_mutex_unlock(&mtx);

// RIGHT — re-verifies after every wake
pthread_mutex_lock(&mtx);
while (!both_dongles_free(coder))
    pthread_cond_wait(&cv, &mtx);
take_both_dongles(coder);
pthread_mutex_unlock(&mtx);

The while makes spurious wakeups harmless: wake, re-check, find the condition still false, sleep again. No special handling, no detection, no distinction between "spurious" and "real but stale" wakeups needed — you simply never trust a wakeup as evidence of anything.

5.3  Three reasons the loop is mandatory

Spurious wakeups get the attention, but they're only one of three independent reasons the while is non-negotiable:

  1. Spurious wakeups — POSIX permits waking with no signal at all.
  2. Stolen wakeups — you were signalled legitimately, but between your wake and your mutex reacquisition, a third thread barged in and took the dongle. Your condition is false again through no fault of the signaller.
  3. Broadcast — everyone wakes, but only one can proceed. The other N−1 must re-check and sleep again; that's the design working correctly.
The unifying principle A wakeup is a hint that the state might have changed. It is never a promise that it did, or that it still has by the time you're running. The condition itself — checked under the mutex — is the only source of truth. This is why the loop isn't defensive programming; it's the correct model of what a condition variable is.
Connect to Codexion An if instead of a while is a classic Codexion bug that survives gentle testing. With 5 coders and generous timings, wakeups are mostly real and mostly unstolen, so it passes. Under the subject's Medium benchmark cases — 800ms cooldowns, heavy queueing, deliberate contention — the stolen-wakeup path fires, a coder takes a dongle another coder already holds, and you've failed the "no dongle duplication" check the evaluation scale explicitly tests for.

Chapter 06

Producer–consumer

6.1  The canonical problem

Why this exists Producer–consumer is the pattern condition variables were essentially invented for, and it's worth learning because nearly every real coordination problem — including Codexion's — is a variation on it. One or more threads produce items into a bounded buffer; one or more consume them. Producers must wait when the buffer is full; consumers must wait when it's empty.

Two waiting conditions, two distinct predicates — which is exactly why it needs two condition variables sharing one mutex.

6.2  The implementation

pthread_mutex_t mtx;
pthread_cond_t  not_full, not_empty;
int buf[N], count = 0;

void produce(int item) {
    pthread_mutex_lock(&mtx);
    while (count == N)                       // buffer full
        pthread_cond_wait(&not_full, &mtx);
    buf[count++] = item;
    pthread_cond_signal(&not_empty);         // a consumer may now proceed
    pthread_mutex_unlock(&mtx);
}

int consume(void) {
    pthread_mutex_lock(&mtx);
    while (count == 0)                       // buffer empty
        pthread_cond_wait(&not_empty, &mtx);
    int item = buf[--count];
    pthread_cond_signal(&not_full);          // a producer may now proceed
    pthread_mutex_unlock(&mtx);
    return item;
}

Every element of this course appears here: one mutex protecting all shared state (buf, count); two condition variables because there are two different things to wait for; while loops on both; and signalling the other side's condition variable after changing the state it cares about.

6.3  Why two condition variables, not one

You could use a single condition variable and broadcast on every change. It would be correct — and wasteful. Every producer would wake every consumer and vice versa, and most would immediately re-sleep (Chapter 3's thundering herd). Separate condition variables per predicate mean a signal only wakes threads that could actually make progress.

Connect to Codexion This maps directly onto a design decision you face. One condition variable for all dongles, broadcast on every release, is simple and correct: every coder wakes, re-checks its own two dongles, and mostly goes back to sleep. One condition variable per dongle is more precise: releasing dongle 3 wakes only the (at most two) coders adjacent to it. The producer–consumer lesson is that the second approach scales better — and with 200 coders, "better" is measurable. Either is defensible; being able to explain the tradeoff is what matters.

Chapter 07

Condition variables in Codexion

7.1  What the subject actually requires

The mandatory part states: dongle state must be protected by a pthread_mutex_t, and "a condition variable (pthread_cond_t) may be used to manage waiting queues." Note may — but then read the fairness requirement: dongles "must grant access according to scheduler," fifo or edf. Together these push you firmly toward condition variables, because polling cannot deliver the latency the 10ms burnout requirement demands, and raw mutexes cannot deliver ordering at all (Course 5, Chapter 4).

7.2  The critical limitation

Read this twice A condition variable gives you no control over which waiter wakes. Not FIFO. Not by deadline. Whichever the implementation picks. So a condition variable, by itself, cannot implement Codexion's fairness requirement. This is the single most important thing to understand about how condition variables fit into your design.

The resolution is a clean separation of concerns:

LayerJobMechanism
MutexProtect the integrity of shared state (dongle flags, the request heap)pthread_mutex_t
Your heapDecide who is next — fifo by arrival, edf by deadlineYour hand-written priority queue
Condition variablePut waiters to sleep efficiently and wake them when state changespthread_cond_t

7.3  The pattern that makes ordering work

The trick is that each coder's predicate isn't just "is the dongle free" — it's "is the dongle free and am I the one the heap says goes next?":

pthread_mutex_lock(&sim->mtx);
heap_insert(sim->queue, make_request(coder));   // register interest, ordered by policy

while (!(dongles_free(coder) && heap_peek(sim->queue) == coder->id))
    pthread_cond_wait(&sim->cv, &sim->mtx);      // wake, re-check, maybe sleep again

heap_pop(sim->queue);
take_dongles(coder);
pthread_mutex_unlock(&sim->mtx);

Now it doesn't matter that the condition variable wakes an arbitrary coder. If the wrong one wakes, its predicate is false (the heap says someone else is next) and it sleeps again. Only the coder the heap selects can pass the while. The ordering policy lives entirely in your predicate — the condition variable is demoted to a pure sleep/wake mechanism, which is all it was ever able to be.

Connect to Codexion — say this at defense "My condition variable doesn't implement fairness — it can't, it wakes an arbitrary waiter. My heap implements fairness, by making 'am I at the head of the queue?' part of every coder's wait predicate. The condition variable just handles sleeping efficiently." That sentence demonstrates you understand both primitives' actual contracts, and it's the design the subject's requirements corner you into once you understand them.
